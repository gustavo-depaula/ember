- **Galleries: one primitive, three displays, two markdown directives (2026-05-23).** Practice/chapter `type: "gallery"` JSON used to be exploded into individual `image` primitives by `apps/app/src/content/preprocessFlow.ts`, dropping the `title` field and orphaning the unused `GalleryBlock` carousel component. Spec'd in `docs/features/galleries.md` and rewired: `GalleryPrimitive` now has `display: 'carousel' | 'stack' | 'row'` (default `carousel`), an optional shared `caption`, optional `weights` for `row`, and items with optional `alt | title | attribution | caption`. The preprocessor passes the primitive through; `GalleryBlock` branches on `display`. Row layout measures its container at runtime ŌĆö if `count * 140px + (count-1)*10px > containerWidth` it switches from CSS-grid to a peek-and-snap FlatList so off-screen bleed becomes the swipe affordance. Books use the same shape via a new marked container-directive extension (`:::gallery`, `:::row`) in `apps/app/src/features/books/markedGalleryExtension.ts`. The extension is the first non-footnote marked plugin in the project ŌĆö pattern for future ones: define a block-level tokenizer with a `start` hook returning the first regex-match index, a `tokenizer` that consumes the full block (and stashes inner caption tokens via `this.lexer.inlineTokens`), and a `renderer` that uses `this.parser.parseInline(tokens)` for those inner tokens. CSS in the reader WebView reads `[data-display]` and `[data-overflow]` attributes (set by JS at measure-time) to pick layout. Touch/click handlers in `paginationScript` short-circuit page-flip when the target is inside `.ember-gallery`, dispatching a `galleryImageTap` message that the React host hands to `ImageViewer`.

- **PDF book ingestion: `My Catholic Faith` (Bp. Morrow, 1949) end-to-end pipeline (2026-05-18).** First book imported directly from a PDF source rather than transcribed HTML/text. The source is a 423-page two-column pictorial catechism with 193 numbered lessons. Pipeline lives at `scripts/import-my-catholic-faith.py` and is fully re-runnable: it downloads the PDF to `.cache/`, wipes `content/books/morrow-my-catholic-faith/{en-US,images}/`, and rebuilds everything from page bounds detected in the PDF itself. Validation: `scripts/validate-my-catholic-faith.py`.

  **Layout-aware extraction beats `pdftotext`.** The PDF has 11pt body text, 9pt commentary expansions, 11pt-bold questions, italic image captions, and a 12pt+ bold lesson heading per chapter. Plain `pdftotext` flattens the 2-column layout in physical order (left-then-right within each line band ŌåÆ reads wrong). The pipeline uses pymupdf's `get_text("dict")`, splits each line into runs of consecutive spans sharing `(size, bold, italic)`, classifies via font reference (`Type3 (428 0 R)` = bold, `Type3 (598 0 R)` = italic, `Type3 (504 0 R)` = regular ŌĆö flags=0 on every span, useless), then sorts left/right columns by `(int(y0 // 7), x0)`. **Floor division, not round, is critical:** `round(y0/5)` for adjacent fragments at y=522.515 vs y=522.488 put them in *different* buckets (105 vs 104) because they straddle the .5 rounding boundary ŌĆö that's what was scrambling bold-question and regular-answer fragments across many lessons. Column split is at x=247 (left col tails end ~245, right col starts cluster at 250+; gutter is sparse at 246-249), not 250 ŌĆö that fix unscrambled the 2-up Q&A layout in lessons 80/82/130.

  **Whitespace artifacts are baked into the PDF text stream**, not extraction-method artifacts ŌĆö `pdftotext`, pymupdf's `dict` mode, and `words` mode all emit `ofthorns`, `ifwe`, `endofthe`, `BlessedTrinity`. Fix: `split_glued(token)` with three paths gated on confidence. (1) CamelCase boundary `[a-z][A-Z]` + nested wordninja per piece; (2) ALL-CAPS path that lowercases, runs wordninja, re-uppercases, handles trailing `'S`; (3) general wordninja with at-least-one-glue-word requirement; (4) binary fallback for tokens with zipfŌēł0 (recovers `ofsin` that wordninja over-splits to `of s in`). Each path validates pieces via `wordfreq.zipf_frequency` (Ōēź3.0, or 2.5/2.7 in the camel+glue and all-caps paths). Tested against 25 cases including proper nouns (`LaRavoire`, `Krishnagar`, `DeVico`) and rare ecclesiastical terms (`Indefectibility`, `Sacramentals`, `Manichaean`) ŌĆö none mis-split. Closed-class glue whitelist + a small `CATHOLIC_VOCAB` (saints, places, biblical verbs) covers the residual long-tail.

  **Section dividers are not chapters.** The printed TOC italicizes ~10 sub-section labels like "APOSTOLICITY OF CATHOLIC DOCTRINES" between numbered lessons ŌĆö visually they look like headings (12pt bold), but they aren't separate chapters. Two consequences: (1) lessons including their following divider can appear as 3-page rather than 2-page spans, which is correct ŌĆö the divider stays with the previous chapter per user direction; (2) when a divider appears at the top of a lesson's first page (e.g. "Eminent Catholics" on lesson 65's page), `find_heading` must search past it for the actual `N. TITLE` line. Solved by tagging non-numbered 12pt-bold lines as `section` (rendered as `## H2`) instead of `heading` (`# H1`), and emitting the H1 first in markdown regardless of physical order.

  **Lesson 61's heading mixes 12pt and 13pt** in the same line ("**61.**" is 13pt, "THE POPE, BISHOPS AND COUNCILS" is 12pt). Since the fragment splitter keys on size, the two halves landed in different fragments; the heading detector saw only "61." Fix: when scanning for the heading line, merge adjacent same-line boldŌēź12pt fragments before pattern-matching.

  **Running-header band y-coordinate is razor-thin.** Page numbers + "MY CATHOLIC FAITH" / running title sit at yŌēł18; lesson 66's heading sits at yŌēł24. Cutoff of y<30 dropped lesson 66's heading entirely. Final rule: only drop y<30 if the line's font is *not* 12pt+ bold ŌĆö a lesson heading near the top of the page is real content, not a header.

  **193/193 chapters, 189/193 header images.** Four lessons (14, 51 originally ŌĆö 51 recovered, 58, 65) and one appendix have illustrations rendered as Type3 vector glyphs which `pdfimages` can't extract. Fallback: rasterize the page region above the heading via `page.get_pixmap(clip=Rect(0, 30, width, heading_y-4))` at 2├Ś zoom and convert via ImageMagick ŌåÆ WebP. Recovered 3 of the 7 originally missing; the remaining 5 are genuinely imageless (the heading sits at the top of the page with no room above for an illustration).

  **Auto-generated TOC mirrors source.** `book.json`'s TOC titles are regenerated on each import run from the extracted lesson headings, so the manifest can't drift from chapter content (Judgement/Judgment, Neighbour/Neighbor, Colors/Colours ŌĆö the PDF uses British spelling intermittently; trying to hand-type a "canonical" TOC produces 21 mismatches against the actual H1s).

  **Lesson 193 + appendix split.** The PDF runs the final lesson + Church Year + Most Important Prayers + Index as one continuous text from page 406 to end. `APPENDIX_OVERRIDE` clamps lesson 193 to pp 406-407; `APPENDICES` carves Church Year (408-411) and Most Important Prayers (412-415) into their own chapter files; the index (p416+) is dropped ŌĆö it's a printed-book navigation aid that doesn't translate.

  **Sub-agent audit loop.** After the initial import, two rounds of sonnet sub-agents (7 + 4 parallel) reviewed chapter batches and surfaced fixable artifact classes beyond the initial ones: glued-on-2-char artifacts (`ofa`, `Iam`), pronoun-glued verbs (`Himselfgave`/`Himselftaught`), space-split words (`Via ti cum`, `dir i ment`, `A men` ŌåÆ `Amen`), bold-question mid-sentence with `?` typeset in regular weight, line-break hyphens not re-joined (`im- pediments`), numeric glue (`of350`), and the `Chruch` source typo. Each got a targeted fix; the regex-driven `SPLIT_FIXES` table feeds into a `PROTECTED_REJOINED` set that prevents `split_glued` from immediately undoing the rejoin (e.g. wordninja sees `Amen` and wants to re-split to `A men`; the guard short-circuits). The `clean_text` second pass runs on the rendered markdown (skipping image-ref lines) to catch typos that only manifest after title-casing converts the all-caps source heading.

- **Content Producers ŌĆö spec'd as a small extension point for runtime-computed content (2026-05-17).** Three real pressures forced the design: (1) the Mass flow is split-brain ŌĆö `packages/mass-of/src/source.ts` owns precedence/Gloria/preface logic in code while `content/practices/mass/flow.json` + 37 fragments mirror those decisions in JSON, so a single liturgical-rule change touches both sides; (2) several books we want (CCC on vatican.va, iBreviary, L├Łrio Cat├│lico, Escriva works) can't be bundled in Hearth for copyright reasons and need runtime fetching with sub-document addressability (cite `book/ccc#507` from a prayer card and either open the reader there or render the paragraph inline); (3) daily snippets like today's gospel, word of the pope, today's office reading want to compose into existing static practices at block level, not replace whole practices. See `docs/features/producers.md` for the design.

  **Three things load-bearing in the design, three explicitly deferred.**

  *Locked in:* (a) **Producer** = built-in code function in its own package, registered by id, returning either `FlowBlock[]` (flow consumers) or cleaned HTML + anchor sidecar (reader consumers) ŌĆö no forced unification. Catalog entry is just `{ id, kind }`; attribution and cache policy live in producer code, not declarative fields. (b) **`include` primitive** in the flow DSL (`{ type: "include", ref: "producer/...", params: {...} }`) usable at any depth ŌĆö same primitive handles whole-flow producers (Mass: one `include` at the root) and composable snippets (gospel-of-the-day inside a morning routine) and anchored book excerpts (`book/ccc#507-509`). (c) **Anchored refs** (`kind/id#anchor[-anchor]`) extend the ref grammar; resolver returns `(chapter, anchorRange)`; reader uses native `scrollIntoView` and HTML subtree extraction. No new rendering pipeline ŌĆö tappable refs work via `<a data-ref="...">` caught by a WebView click handler.

  *Deferred:* (a) **Declarative-recipe runtime** ŌĆö premature; extract only after 3-4 built-in producers converge on the same shape. (b) **Reader IR / unified Block type** ŌĆö today's WebView+HTML is sufficient; IR pays off only when we want multiple render targets or structured highlights. Doing it later is cheap because IR would be internal. (c) **Web parity for external fetching** ŌĆö CORS proxy vs. native-first decision deferred until a real user need bites.

  **What the user vetoed during planning (worth remembering for future similar designs):** I initially proposed an IR refactor as a prerequisite, a unified `Block` type across flow and reader, declarative adapter recipes from day one (`html-extract` / `json-api` / `widget-embed` kinds), a `producer` field on `PracticeManifest` as a whole-flow swap, and producer manifests carrying `{ attribution, license, lawfulUse, cachePolicy }`. Each was overengineering. The user pushed back: producers compose at block level not practice level (otherwise you can't drop today's gospel into a morning routine), built-in code first (declarative recipes only when shared patterns prove themselves), HTML is sufficient (IR is internal and can be added later cheaply), and config fields belong inside producer code (built-in producers don't need configuring). The final spec is much smaller ŌĆö one doc, one extension point, two small additions to existing data shapes (`source`/`anchors` on `BookEntry`, `include` in the flow DSL).

  **External cache stays on-device.** The legal load-bearing decision: Hearth never holds third-party bytes. Cache lives in a new `external_content` SQLite table on each device, with the existing 200MB LRU at `apps/app/src/content/store.ts:204-231`. Even transient server-side caching arguably crosses from "Safari Reader" into "republishing" for the copyrighted sources, so we don't.

- **Merged `prayer` and `practice` into a single `practice` kind (2026-05-15).** Two reasons: scheduling a Pater Noster at 3pm was impossible (only practices were schedulable), and the data shape was already 95% the same ŌĆö a prayer's `body: FlowSection[]` is functionally just a flow's `sections`. Migration moved ~99 `content/prayers/*.json` into `content/practices/<id>/manifest.json` with inline `flow: { sections: <old body> }` ŌĆö most short prayers now skip `flow.json` entirely.

  **The resolver alias is the key trick.** All 1,099 existing `{ type: 'prayer', ref: 'our-father' }` refs in practice flows keep working unchanged because `fetchPrayerSync(id)` now passes `'practice'` as the hint to `canonicalize()`, which finds `practice/our-father` and returns its `flow.sections` as the prayer body. The `prayer` section type in the flow DSL is preserved (it's a rendering hint ŌĆö collapsible block with title); only the catalog kind disappeared. `resolveCanticle` works the same way; the 3 canticles (Magnificat / Benedictus / Nunc Dimittis) keep their `subtitle` / `source` as optional `PracticeManifest` fields.

  **Five name collisions** existed where the same id had both a `prayer/` and a `practice/` entry (act-of-reparation, anima-christi, memorare, regina-caeli, suscipe). The practice was a ritual wrapper (sign-of-cross ŌåÆ prayer text ŌåÆ sign-of-cross) and the prayer JSON held the canonical body. Resolution: substitute the prayer's body inline where the practice's flow had a self-ref or held a duplicated text block, then convert the practice's `"flow": "flow.json"` legacy string to inline `flow`. See `scripts/migrate-prayers-to-practices.py`.

  **Collections** got a one-shot rewrite: 128 `ref: "prayer/<id>"` ŌåÆ `ref: "practice/<id>"` across 11 JSONs. Refs *inside* practice flows stayed bare and Just Work via `canonicalize`.

  **UI fully erased the distinction:** `PrayerModal`, `EssentialPrayersRow`, `AllPrayersList` deleted. Browse, search, and collection ItemCard collapse to one render path ŌĆö every item opens a practice page. The hardcoded "Essential Prayers" curation is now a separate concern (can be rebuilt as a featured collection if we miss it).

- **Creator metadata pre-computed by the deploy workflow, not the client (2026-05-14).** The Catholic Creators feature had two expensive YouTube discoveries running per device, per refresh: scraping the channel page's `og:image` to get a creator avatar (Atom feed has no channel image), and HEAD-probing `/shorts/<videoId>` for items in the channel feed that weren't in the UUSH playlist (UUSH only surfaces the last ~15 shorts, so older shorts in the channel feed get misclassified as long-form videos until you probe). For 20 creators ├Ś ~15 ambiguous items each that's ~300 HEAD requests per device per refresh. Also: YouTube has started rate-limiting `/feeds/videos.xml?playlist_id=ŌĆ”` calls aggressively per-IP ŌĆö a single dev workstation can easily land in a 404 hole after a few minutes of testing.

  Solution: do the discovery once, in CI, on a fresh runner IP, write a static JSON file per creator into the Hearth corpus, and have the client just consume it.

  **Pipeline:** `scripts/build-creator-meta.py` runs after `build-corpus.py` in `.github/workflows/deploy.yml`. Walks `content/creators/*/manifest.json`; for each YouTube channel scrapes the og:image and discovers shorts (UUSH + HEAD-probe of ambiguous channel-feed items, parallelized via `concurrent.futures.ThreadPoolExecutor`); for each podcast/RSS feed extracts the channel-level `<itunes:image>`/`<image>`/`<icon>`. Writes `_site/hearth/v2/creator-meta/<slug>.json` with `{creatorId, generatedAt, channelImage, shortVideoIds[]}`. Stdlib only ŌĆö no extra Python deps in CI.

  **Schedule:** the workflow now also fires on `cron: '15 * * * *'` (hourly at :15). The Pages deploy is fast since the existing corpus is mostly cached; only the creator-meta files change between runs. Shorts classifications lag at most an hour for new uploads ŌĆö acceptable since a video never *changes* kind once published.

  **Client:** `apps/app/src/features/creators/feeds/fetcher.ts` tries `hearthUrl('creator-meta/<slug>.json')` at the start of each refresh. When present, the YouTube branch skips the og:image scrape, the UUSH fetch, AND the per-video HEAD probe ŌĆö uses the prebuilt `channelImage` + `shortVideoIds` directly, only fetching the channel_id Atom feed for the current item list. When absent (e.g. a creator added between CI runs), the live discovery path runs as a fallback. The `defaultPrebuiltLoader` dynamic-imports `@/lib/hearth` ŌĆö that module transitively pulls in `expo-sqlite` via the cache repo, which vitest/jsdom can't parse; the dynamic import means the fetcher tests don't touch it as long as they inject their own loader (typically `async () => null`).

  **Why this instead of a Cloudflare Worker / Edge proxy** (the journal-noted TODO at `apps/app/src/features/creators/feeds/fetcher.ts`'s `feedUrl()` for CORS): operational simplicity. The Pages deploy is already running on each content change; layering one more Python step that costs ~10s in CI and emits ~20 small JSON files is strictly free. A Worker remains the right answer if/when we need real-time freshness or a JSON-normalized feed API; for now, a CI-built static pack covers the actual pain (avatar discovery + shorts classification) without standing up new infrastructure.

- **Spiritual threads ŌĆö daily-only resolution loop shipped (2026-05-13).** Phases 1ŌĆō3 of `docs/features/spiritual-threads/` plus a daily-only Plan-of-Life Resolutions panel and the petitionŌåÆthanksgiving bridge from 6a. Phases 4 (higher-level pieces), 5 (periodic reviews), and the `'liturgical-event'` schedule rule are **deferred** ŌĆö non-daily windows had real defects (post-Pentecost OT collapse from `addDays(start, 100)`, missing logical-day cutoff) and we chose to defer over partial fix. PRD `04-plan-of-life.md` / `05-periodic-reviews.md` are marked deferred and remain the design-of-record.

  **Architecture decisions worth remembering:**

  - **One Movement projection, two event streams.** Petitions and thanksgivings share `Map<id, Movement>` discriminated by `kind`, but their events are distinct (`IntentionRaised`/`Answered`/`Expired`/`Retired` vs `ThanksgivingOffered`/`Updated`/`Retired`). Hooks expose kind-bound wrappers (`useActiveIntentions`, `useActiveThanksgivings`); selectors share a `matching()` generator.
  - **Cadence-aware closure.** `state: 'active' | 'closed'` is the boolean every query needs; `closure_kind: 'answered' | 'expired' | 'retired'` is the analytic detail. Action-menu verbs gate on `cadence` (`perpetual` ŌåÆ only "Retire"; `goal` ŌåÆ "Mark answered"; `bounded` ŌåÆ both + auto-expiry).
  - **Bounded petition expiry uses sweep + lazy fallback.** `useExpirySweep` fires on app foreground + 1h interval and emits `IntentionExpired`; read hooks defensively skip overdue active items so a missed sweep doesn't show stale state. Pure logic in `findExpired.ts` so vitest doesn't have to load `react-native`.
  - **Engine stays pure.** `EngineContext.resolutions` / `windowFor` / `supportsMovements` are read-only injections. The four new DSL primitives (`offering`, `capture-movement`, `capture-resolution`, `review-resolution`) resolve to rendered shapes; **writes happen in the player**, not the engine. `withSpiritualThreads(ec)` is the host wiring that snapshots Zustand store state at resolve time and closes over the host's "today" for `windowFor`. Lives in `apps/app/src/content/engineContext.ts`.
  - **Engine doesn't know about clocks.** `ec.windowFor(level, forward)` takes only level + direction; the host closes over its anchor. There is no `ec.logicalDay()` ŌĆö the engine asks "what window?", the host decides what "today" means.
  - **Offering is in-flow only.** No separate `OfferingPickerSheet`; the `offering` block IS the picker. Renders the active intentions (or a quick empty-state CTA) inline at the liturgical moment the practice asks for them.
  - **Flow resolution is a snapshot.** When a user captures a daily resolution mid-Examen, the *running* flow doesn't reshuffle ŌĆö `withSpiritualThreads` reads state once at resolve time. Backing up to a prior block shows the same data the user saw on first pass.
  - **Capture-resolution pre-fills from the projection.** `ec.resolutions.inWindow(level, starts_at)` looks up an existing resolution whose window matches the block's target window. If found, the block opens in display mode with a Change affordance instead of an empty form; tapping Change reopens the form pre-filled and `Save` calls `reviseResolution` (preserves id, emits `ResolutionRevised`). Stops users from accidentally creating duplicate resolutions across multiple Examen runs in the same day.
  - **No "Skip" controls.** Both `capture-movement` and `capture-resolution` are *optional by default*; the way to skip is to not submit. An explicit Skip button would be irreversible (once tapped, the form is gone), which is hostile when the user just wants to scroll past. The `optional` prop was removed entirely from the DSL.
  - **Explicit-open everywhere.** Capture forms (offering, capture-movement) open only on a "+ Add" tap. Auto-opening a textarea the user didn't ask for is the same anti-pattern; one consistent fix.
  - **One "today" everywhere, with a 4am cutoff (configurable).** `useToday()` / `getToday()` apply `logicalDay(new Date(), 4)` in live mode ŌĆö a new day doesn't start until 4am. Examen done at Wed 1am still reviews R(Tue) and writes R(Wed); the day carousel still shows Tuesday; completion stamps still go to Tuesday. The resolutions / day carousel / completions / wall all agree on what day it is. Time-travel honors the user-picked date as-is (no cutoff applied ŌĆö picking May 12 means May 12, not "depends on the hour"). The journal previously documented an asymmetry where only resolutions honored the cutoff ŌĆö that was rejected in favor of consistency.
  - **Resolution lifecycle: window stored, state derived.** A `Resolution` has `starts_at` / `ends_at`; "active" / "closed" are computed from `now` vs window. Reviews are separate (`ResolutionCheckin` accumulates N times; `ResolutionReviewed` is the canonical close). `pending-<level>` = most-recent unreviewed; `active-<level>` = the one whose window contains now.
  - **Two checkin events vs one with a mode field.** `ResolutionCheckin` and `ResolutionReviewed` are separate events because their cardinality differs (N vs 1) and only `ResolutionReviewed` marks a resolution as canonically closed. A resolution with three Checkins but no Reviewed is still pending.
  - **`ResolutionLevel` is `'daily'`.** The type union is intentionally a singleton to keep the call sites typed for when non-daily resumes. `REVIEW_TARGETS` in `engine/resolve.ts` only registers `'active-daily'` and `'pending-daily'`.
  - **`group` DSL block for conditional sections.** `{ type: 'group', skipIfEmpty: true, sections: [...] }` ŌĆö collapses entirely when its children resolve to only structural primitives (subheading / divider / heading). Verificatio in the Examen wraps subheading + review-resolution + divider this way: on first run with no prior resolution, the whole section disappears cleanly instead of leaving an orphan header. Useful anywhere a chrome-only section needs to gate on a single conditional child.

- **CCEL pilot conversion learnings (2026-04-28).** Ran the importer against all four pilot ThMLs end-to-end on a machine with CCEL access. Final shipped library: `ccel-classics-0.1.0.pray` (1.5MB, 656 files, 4 books / 638 chapters total). Several real-world quirks the synthetic fixture missed:

  (1) **DC fields use both element and HTML form**: CCEL ThML 1.04 uses `<DC.Creator scheme="short-form">Thomas ├Ā Kempis</DC.Creator>` (element), not the HTML `<meta name="DC.Creator" content="ŌĆ”"/>` form. The original extractor only looked at `<meta>` and pulled nothing. Fixed `_meta_value` to scan element form first, then HTML; added `_meta_values_with_attr` to disambiguate by `scheme`/`sub`. Author preference: `scheme="short-form"` ŌåÆ `scheme="file-as"` ŌåÆ first DC.Creator. Skip `scheme="ccel"` (the bare slug).

  (2) **`<DC.Date sub="Created">` is the digital-edition timestamp, NOT composition**. Imitation showed `1998-12-10`, Devout Life `2002-08-08`. Importer now prefers `<firstPublished>` / `<published>` from `<generalInfo>`; only accepts `DC.Date` if it's a pre-1900 4-digit year. For modern translations / works without `firstPublished`, pass `--composed` explicitly.

  (3) **Source URL fallback**: not every CCEL file has a `DC.Identifier` URL; many only have the slug `ccel/kempis/imitation.htm`. Synthesize `https://www.ccel.org/ccel/<authorID>/<bookID>/` from `<authorID>`/`<bookID>` in `<electronicEdInfo>` when no http(s) DC.Identifier exists.

  (4) **`DC.Language` may be ISO 639-3** (`eng`, `por`, `lat`, `grc`). Extended `_LANG_MAP` to handle 3-letter codes.

  (5) **Skip CCEL navigation indexes/contents**: every CCEL file ends with a `<div1 title="Indexes">` (or `<div1 title="INDEX.">` for older files) wrapping `<div2 title="Subject Index">`, `<div2 title="Index of Scripture References">`, etc. Some also start with a print-edition `<div1 title="CONTENTS.">` table of contents. These are auto-generated reference data and produce mangled output. Filter by title (`index`, `indexes`, `contents`, `index of ŌĆ”`) at root level + at chapter level ŌĆö covered by `_is_index_section`.

  (6) **CCEL chapters open with print decorations**: `<h4>The First Chapter</h4>` then `<h3>{title}</h3>` before the first paragraph. We already inject our own `# {title}` H1 from the section attribute, so any leading sub-heading inside a chapter body is redundant ŌĆö strip everything until the first non-empty, non-`#` line.

  (7) **`<p> </p>` whitespace-only spacers** are common in CCEL files for visual layout. Drop them entirely instead of emitting blank "  " lines.

  (8) **Whitespace doubling around `<pb/>`**: `<p>ŌĆ”about\n\n<pb/>\n\nthe TrinityŌĆ”</p>` produced `learnedly about  the Trinity` (double space). Added `re.sub(r" {2,}(?!\n)", " ", body)` in `render()` ŌĆö collapses 2+ spaces to 1 except when they form a Markdown hard break (`  \n`).

  (9) **`<verse>` in Augustine's Confessions** is a poetic line group (children: `<l>`) ŌĆö same as `<lg>`. Added to the `lg/verse` handler.

  (10) **Recursion bug in unknown-block fallback**: `<table>` was in `_BLOCK_TAGS` but had no handler in `_walk_block`, so it fell through to a `_walk_inline(el, b)` call that routed back to `_walk_block` based on `_BLOCK_TAGS` membership ŌåÆ infinite recursion. Fixed by walking children directly in the fallback rather than re-dispatching the same element.

  (11) **CCEL `<div1 type="Book" title="Preface">`**: Summa Contra Gentiles has a Preface div with `type="Book"`. Original slug logic preferred `type` over title, producing `book.md`. Re-prioritized: title takes precedence over bare type when no `n` is present.

  (12) **Long titles ŌåÆ filesystem errors**: Devout Life's `<div1>` titles include the full part-summary, then chapter titles add another sentence ŌĆö concatenated, the slug exceeded 255 bytes (macOS filename limit). Slug now detects "PART/BOOK/CHAPTER N" prefixes and uses just the prefix (`part-i`, `chapter-vii`); falls back to the first six words of the title.

  (13) **Augustine Confessions has no `<div2>`** ŌĆö the whole text is `<div1>` per book with `<p class="chapter">CHAPTER I</p>` paragraph-markers inline. `--chapter-level auto` correctly picks `div1`, producing 13 large book-length files (43ŌĆō150KB each). The print-edition CHAPTER markers stay as plain paragraphs in the output; promoting them to `## H2` based on the `class="chapter"` attribute is a future refinement.

## Architecture Decisions

- **Unified flow system replaces variants, forms, and multiple flows (2026-04-12).** The content engine accumulated four content-selection mechanisms (variants, forms, multiple flows, setKeyOverride) each built for specific practices. Decision: collapse all four into one `select` section type ŌĆö a conditional branch that reads context (day of week, time of day, user preference) and picks an option. One flow per practice. Different meditation traditions become separate practices grouped by `alternativeTo`. Full spec: `docs/features/unified-flow-system.md`. Key insight: all four mechanisms were just conditional content selection with different indirection layers. The `variant` DB column was never actually written ŌĆö every user was on the default.

- **DSL completeness: no practice should need custom app code (2026-04-12).** The flow JSON must be expressive enough that the app is a pure reader. The `useLiturgicalMeditation` hook (118 lines, meditacoes-ligorio only) was the clearest violation ŌĆö calendarŌåÆchapter resolution hard-coded in React. Fix: `resolve` field on `FlowDefinition` with pluggable strategies (first: `"liturgical-day"`), and dynamic `prose` sections that load book chapters by template-variable chapter IDs. The resolution algorithm stays in `@ember/liturgical` but is invoked declaratively from the flow JSON, not from a practice-specific hook.

- **Visible `select` blocks require explicit app-side rendering + override wiring (2026-04-14).** After migrating practices to unified flow, the "title-only, no body" symptom appeared only on practices whose main content lived under top-level visible `select` sections (e.g., Mass). The content engine was already emitting rendered `select` sections, but `SectionBlock` had no `select` case, so those branches rendered as `null`. Fix: add `SelectBlock` UI renderer and thread `selectOverrides` state through `PracticeFlow` so selector changes re-resolve flow content live. Silent/static patterns (e.g., Way of Light, Bible+Catechism) were unaffected.

- **Large-flow `select` switching is fast only when non-selected branches are lazy (2026-04-14).** For visible `select`, eagerly resolving sections for every option on each pass caused noticeable delay on heavy flows like Mass. The UI only needs labels for all options and sections for the active option, so the engine now resolves sections only for the selected option and emits empty section arrays for non-selected options. This keeps behavior identical while removing most of the per-toggle work.

- **Unified flow completed migration path for dynamic Liguori meditations (2026-04-14).** The old `useLiturgicalMeditation` hook was removed and replaced with declarative flow resolution: `resolveFlowAsync` executes flow-level `resolve` strategies (starting with `liturgical-day`), injects template vars, preloads dynamic prose chapter text, and then renders sections without practice-specific React logic. This made Liguori content a pure flow JSON concern and aligned slot naming previews to show the first visible select choice in Plan of Life.

- **Legacy slot flow/variant plumbing removed from runtime writes (2026-04-14).** Slot creation/update no longer writes a `variant` field, and the unused `changeSlotFlow` mutation path was removed from repositories/hooks. The database column remains for backward compatibility, but runtime behavior now treats slots as practice-scoped schedules with unified-flow selection handled by the engine.

- **Dynamic prose must support book-language fallback to avoid false-empty renders (2026-04-14).** `meditacoes-ligorio` is PT-BR-only content, so strict active-language loading can render empty prose when app/content language is EN. The fix was to make dynamic prose language behavior explicit (`langPolicy`) and preload chapter text across candidate languages (active/content/book languages), then localize from the merged payload. This keeps the flow declarative while keeping side-effect handling inside bounded source adapters.

- **Flow contract now supports explicit versioning (`flowVersion`) (2026-04-14).** The engine accepts legacy flows with no version and validates `flowVersion: "1"` for explicit-contract flows, while rejecting unsupported future versions early. This gives us room to evolve semantics safely without silent behavior drift.

- **Content loader migration phase used canonical `flow.json` preference without practice IDs (2026-04-14).** During unified-flow rollout, filesystem loading preferred top-level `flow.json` over legacy manifest declarations to avoid practice-specific branches. This transitional behavior was later tightened into strict `manifest.flow` loading once content migration was complete.

- **Legacy flow migration paths removed from filesystem loader (2026-04-14).** The temporary runtime adapters (`migrateLegacyLiturgicalMeditationFlow`, `flows[]` manifest fallback, canonical-vs-declared flow scoring) were removed. Loader contract is now strict: every practice must declare `manifest.flow`, and content was migrated accordingly.

- **Legacy Liguori sidecar flow removed (2026-04-14).** `practices/meditacoes-ligorio/flows/daily.json` was removed to avoid split truth and stale fixed-slot placeholders (`feastLabel`, `meditation-feast`). Canonical `flow.json` is the single source of truth.

- **Liguori resolver now includes monthly supplemental meditations on day 25 (2026-04-14).** `resolveLiturgicalDay` emits supplemental entries for the 25th-day monthly set, and the content-engine strategy includes those chapters in the `meditations` array. This restores expected multi-meditation days on the 25th while keeping logic in the liturgical kernel instead of app-layer hooks.

- **Day-25 rendering fix required reserve fallback for temporal holes (2026-04-14).** Some 25th dates (e.g., during Pentecost-transition gaps) had no temporal chapter in the map, so only the monthly supplemental meditation appeared. `resolveLiturgicalDay` now falls back to `reserves` when temporal lookup fails, preserving the expected ŌĆ£temporal + monthly 25thŌĆØ pairing.

- **Liguori recurrence keys now use generic `weekdaysOfMonths` + fixed-date keys (2026-04-14).** The old family keys (`first-friday/*`, `march-wednesday/*`, `25th-of-month/*`) were migrated to two generic mechanisms: fixed-date keys (`MM-DD`, e.g. `04-25`) and `weekdaysOfMonths` keys (`1st-friday-of-january`, `3rd-monday-of-february`). ## Content & Licensing

- **DRB is the only viable bundled English Catholic Bible.** The Douay-Rheims (1749ŌĆō1750) is fully public domain. Every other major Catholic English translation (NABRE, RSV-CE, NJB, etc.) is under active copyright. Source: `xxruyle/Bible-DouayRheims` on GitHub, MIT license.

## APIs & Integrations

- **Bolls.life API: free, no auth, generous rate limits.** Base URL: `https://bolls.life`. No documented rate limits, but don't abuse it. Responses cached indefinitely in SQLite since Bible text never changes.

- **EF propers bundled from Divinum Officium, not Missale Meum API.** Missale Meum wraps DO data but only supports English + Polish. We need Portuguese too, so we parse the raw DO files directly. The bracket-based format (`[Introitus]`, `[Oratio]`, etc.) is parsed at build time by `scripts/parse-do-propers.ts` into JSON. Portuguese coverage is partial (207 vs 480 Tempora files) ŌĆö the app falls back to English via `localizeContent`. Cross-references (`@Tempora/...`, `@Commune/...`) point to `missa/` or `horas/` Commune directories. Prayer macros (`$Per Dominum`) expand from `Ordo/Prayers.txt`. The Commune files for Mass sections live in `horas/{lang}/Commune/` (same files serve both Office and Mass).

- **Evangelizo: OF reading text but with quirks.** `https://feed.evangelizo.org/v2/reader.php?date={YYYYMMDD}&lang={LANG}&type=all` provides full daily reading text. Limitations: response may contain HTML that needs stripping, date range limited to ~30 days from current date, no CORS headers (may need proxy for web), and does NOT provide collects/antiphons.

- **Catholic Readings API: OF calendar + references only.** GitHub Pages hosted (free, MIT). Provides Scripture references (not full text) and liturgical day metadata. - **Universalis JSONP: potentially has OF propers but unreliable.** Claims to provide full Mass propers via JSONP. Documentation is vague and incomplete. Legacy JavaScript callback pattern doesn't fit React Native well. Worth investigating further but don't count on it.

- **Bible Gateway and API.Bible are not viable.** Bible Gateway discontinued its API and prohibits scraping. API.Bible has FUMS tracking requirements implying online-only usage, unclear Catholic translation availability.

## Technical

- **ProperSlot retries share one OF query key.** Each slot on the Mass screen (Introit, Collect, Antiphon, etc.) has its own `useProperForSlot(slot, 'of')` call, but all slots read from a single `['of-propers', dateKey, language]` query. When a user taps "Try again" on any failed slot, `ofQuery.refetch()` triggers one network request and TanStack Query re-derives every slot's section from the refreshed data ŌĆö no fan-out, no per-slot request storm. Same approach for EF but each slot has its own key (EF reads from bundled/Hearth propers, independent slots).

- **Global mutation error handler uses TanStack Query's MutationCache, not per-hook wiring.** Across 39+ mutations in 10 features, none had `onError` handlers ŌĆö failures were silent. Rather than retrofitting each hook, `_layout.tsx` configures `QueryClient({ mutationCache: new MutationCache({ onError }) })` once. The global handler opens `ConfirmSheet` in single-action mode with `i18n.t('error.somethingWrong')` + the error message (or `error.tryAgainLater` fallback). Per-mutation `onError` still wins: the global handler guards with `if (mutation.options.onError) return`. Trade-off: every mutation error now shows a dialog unless explicitly suppressed ŌĆö acceptable because the fallback is far better than silent failure.

- **The original `getLiturgicalSeason()` was an antiphon scheduler, not a season calculator.** Its boundaries mapped to the Marian antiphon switching dates (AdventŌåÆFeb 1, Feb 2ŌåÆHoly Wednesday, etc.), which don't correspond to actual liturgical seasons. The antiphon schedule is a separate traditional system with its own date ranges ŌĆö decoupled from seasons in the rewrite.

- **OF and EF seasons share Easter and Advent but diverge everywhere else.** Key differences: EF has Epiphanytide (Jan 14 to Septuagesima) and Septuagesimatide (3 pre-Lent Sundays using violet), neither of which exist in OF. EF has "Time after Pentecost" where OF has "Ordinary Time." Christmas ends at Baptism of the Lord (OF, ~Jan 11) vs Octave of Epiphany (EF, Jan 13). A single `LiturgicalSeason` union with 8 values covers both forms cleanly ŌĆö each form uses a subset.

- **Year-boundary edge case in season calculation.** Dec 25 is after the start of Advent, so a naive "if date >= Advent start, return advent" incorrectly classifies Christmas Day. The fix: check Christmas (Dec 25+) before checking Advent. Similarly, Jan 1 is still Christmastide in both forms.

- **Divine Office migrated to practice content format.** The custom engine (`divine-office/engine.ts`) and `PrayerFlow` component were replaced by JSON flow files + the shared content engine. - **Friday abstinence rules differ significantly by jurisdiction.** Universal canon law (Canon 1251) requires meat abstinence on ALL Fridays year-round. - **PDF book imports ŌĆö extraction technique that actually works (2026-04-25).** The `scripts/parse-aquinas-catechetical.py` workflow is reusable for any text-based (non-scanned) Catholic-classic PDF. Three quirks worth knowing:
  - **Hybrid extraction.** `pdftotext -bbox-layout` emits XHTML where each `<block>` is a paragraph ŌĆö *the* signal `pdftotext -raw` and default mode both lose. Use `-bbox-layout` for body, but `-raw` for any 2-column endnote section: `-bbox-layout` reads the columns interleaved (1, 11, 2, 13, 3, 14, ŌĆ”) while `-raw` linearizes them top-down per column (1, 2, 3, ŌĆ”, 17). Parsing endnotes off the bbox output and sorting by N is also workable but loses multi-line note continuity.
  - **Form feeds glue page numbers to text.** `pdftotext -raw` joins page-break footers to the previous token via `\x0c` ŌĆö `ENDNOTES66` (note 66) and `Penance.66` style glitches. Strip `\x0c` from raw text before any line-based regex.
  - **Python 3.14 ships with a broken pyexpat on this machine** (`Symbol not found: _XML_SetAllocTrackerActivationThreshold`); ElementTree raises on import. Use `python3.13` for any XML parsing in import scripts. The shebang `#!/usr/bin/env python3.13` is the safe default for poppler-XML pipelines.
  - **PDF heading detection by ALL-CAPS works for ~80% of cases.** Inline sub-section headings buried mid-block (e.g. `"HAIL MARY"` and `"FULL OF GRACE"` quoted ALL-CAPS subheaders inside the Hail Mary chapter) survive as inline quoted phrases. They're readable but not indexed as `##`. Acceptable for v1; revisit if a future book has many of them.
  - **Scholarly editions can mix footnote conventions.** Collins's body uses `[N]` brackets; his Translator's Preface uses bare-superscript style that pdftotext renders as digits attached to a year (`year 1225.1 The` for `year 1225.[1]`). One regex (`\b\d{4}[.,]\d{1,2}\s+[A-Za-z]`) caught both instances in this PDF.
