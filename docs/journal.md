- **Residual post-load interaction hangs: the in-session catalog re-fetch re-downloaded + re-parsed + re-wrote 522KB every launch; fixed with a conditional GET, and the book warms were deleted (2026-07-04).** After PR #278's three fixes, taps/scroll still stuttered for a beat right after boot. Culprit: the `runAfterInteractions` background refresh (`_layout.tsx`) calls `loadCatalogFromHearth({networkFirst:true})`, which unconditionally downloaded `catalog.json` (**522KB, ~1930 items, growing**), `JSON.parse`'d it, and `setCache`'d it (`JSON.stringify` + SQLite write) **on every launch even when nothing changed** — tens of ms of JS-thread work in the interaction window.
  - **Fix = HTTP conditional GET in `fetchHearth` (`lib/hearth.ts`).** GitHub Pages serves a strong ETag (via Varnish) and honors `If-None-Match` → **304 with no body** when unchanged, so we skip transfer + parse + stringify + write entirely. ETag is stored in the **cache table** under `${key}:etag` (not preferences — `setPreference` fires a cross-tab `broadcastChange`; and the `hearth:` prefix means `clearCache('hearth:')` already wipes it on a hearth switch). Guards: branch on `res.status === 304` *before* `!res.ok` (304 isn't ok); on a 304 with an empty body cache (etag outlived the body), re-fetch once unconditionally. `conditional` is off for `isLocal` dev. Generic in `fetchHearth`, so any mutable named file benefits; hash-addressed blobs (`getJson`) are immutable and don't need it.
  - **The boot "warms" for external books were unnecessary background work.** `warmCccBooks()` deleted — its manifests are static/synchronous, so `registerCccCatalog()` now `rememberManifestBody`s them inline at registration (resident immediately for sync cross-ref anchors; on-demand resolver kept as fallback). `warmEscrivaBooks()` deleted — it fetched **28 TOCs from escriva.org fire-and-forget every boot**; opening a book already resolves the manifest async via TanStack Query → `ensureManifestBody` (book details `index.tsx`). The one gap: `BookReader` read `getBookEntry()` **synchronously**, so a deep link straight into the reader (no details screen first) relied on the warm — fixed by making `BookReader` resolve its manifest via `useQuery`+`ensureManifestBody` with the resident manifest as `initialData` (instant on normal nav, async only on cold deep link). `BookReader.tsx` cross-ref reads stay sync — Hearth books are covered by `warmDeferredManifests`, CCC is now resident, and nothing cross-refs Escrivá books.
  - **Root cause under all book warms:** `getBookEntry → residentItem → getRememberedManifest` is sync/resident-only with no fetch fallback. Anything that needs a book manifest synchronously forces a warm to exist. The durable pattern is to resolve referenced manifests on the async path (`ensureManifestBody`) at the point of use, not warm everything at boot.

- **CCC + Compendium became external books scraped live from vatican.va; the bespoke catechism reader was deleted (2026-06-19).** The full Catechism used to be a third-party `ccc.json` (English-only) read by a custom 3-pane reader (`features/catechism/`, `stores/catechismStore`, `lib/catechism.ts`). Both are gone. Now `book/ccc` and `book/compendium` are **external books** (`BookEntry.source = { type:'external', producer:'producer/ccc-book' }`, same mechanism as the Escrivá books) registered at boot by `content/cccCatalog.ts`, read with the normal book reader, scraped from vatican.va on demand and cached per-device in `external_content`. Hard-won gotchas:
  - **The full CCC IS on vatican.va in Portuguese** — at `archive/cathechism_po/` (note the misspelling "cathechism", same quirk as the Compendium's `_po`). The CCC index (`archive/ccc/index.htm`) lists every language. Don't trust the archive landing page, which only shows languages for the Compendium.
  - **EN and PT are paginated completely differently.** PT = one clean page per chapter, paragraph range in the filename (`p1s1c1_26-49_po.html`) → no crawl. EN (`ENG0015/__P*.HTM`) = ~374 tiny IntraText pages (~9 paragraphs each), base-36 codes (P1…PZ,P10…) chained by "Next" links, **no** paragraph→page map on the site. We crawl the Next-chain once (`scripts/build-ccc-en-index.mjs`) into `sources/ccc/en-pages.json`; the text is frozen (HTML dated 2003).
  - **One canonical TOC, shared across languages** (the reader enforces a single TOC; chapters keyed by `pointRange`/paragraph). We adopt the PT chapter divisions (27 leaves, §1–2865 contiguous, in `sources/ccc/structure.ts`). EN pages straddle chapter boundaries, so a chapter's assembled HTML is `trimToRange`'d to drop leaked neighbour paragraphs — but far-out-of-range embedded numbers (the Ten Commandments enumerated 1–10 at §2051) are kept-but-untagged, not dropped.
  - **IntraText markup is maddeningly inconsistent.** "Next" links are `<a href=__P3.HTM>Next</a>` early and `<A\nhref="http://…__P86.HTM">Next</A>` later. Paragraph numbers are bare (`<p class=MsoNormal>1 …`), bold (`<b>`), or italic with a style attr (IN BRIEF: `<p class=MsoNormal><i style='…'>2857`). The crawler/parser must allow attrs on `<i>`/`<b>` and filter candidate numbers by **monotonic sequence** (else a page enumerating the Ten Commandments looks like it contains "paragraphs 1–10" and gets range `[1,2051]`, polluting the prologue).
  - **The PT source has real typos:** §2217 is printed "2117", §2439 is "1439" (digits transposed). The text is present, just mis-numbered. Two paragraphs out of 2865 — not worth chasing. EN has one merged paragraph (§2077 shares a `<p>` with §2076). Net: completeness is essentially perfect.
  - **Cross-references work** — the Compendium's answers emit `<a data-ref="book/ccc#1-25">`; the foliate WebView (`bootstrap.raw.js`) now posts `data-ref` taps, and `BookReader` resolves `book/<id>#<anchor>` via the target book's `anchors` index (jump within book, or `router.push` across books). Re-run `node bundle.mjs` in `features/books/reader/foliate/` after editing `bootstrap.raw.js`.
  - **`setManifestBodyResolver` was single-slot** — made it an ordered list so Escrivá + CCC resolvers coexist.

- **Bespoke holy cards moved out of app code into a Hearth-served data file (2026-06-18).** Adding a saint card used to mean editing TypeScript in three places — `features/saints/data/saints.ts` (the id + image + i18n-key array), `data/catalog.ts`'s `bespokeFeast` map, and a 4-key `saints.<id>` block in **both** locale files — then shipping an app release. Now the whole catalog is `content/practices/saint-of-the-day/data/holy-cards.json` (`{ version, cards: [{ id, feast:{month,day}, name, patronOf, prayerExcerpt }] }`, strings as `{ "en-US", "pt-BR" }` `LocalizedText`). Adding a card = append an object + drop `content/saints/<id>.webp` + `pnpm build:corpus`. No app code, no release. Gotchas:
  - **It rides the existing saint-of-the-day practice, not a new corpus kind.** The build already emits any `data/*.json` a practice declares in its manifest `data` block as a hash-addressed `dataHashes` entry keyed by logical name (see `build_practices`). So registering `"holy-cards": "data/holy-cards.json"` in the manifest was the entire build-side change — `useHolyCards.ts` fetches it the same way `useSaintOfDayIndex.ts` fetches the day index (look up `practice/saint-of-the-day` → `getRememberedManifest` → find the named `dataHashes` ref → `getJson`). New corpus *kinds* need Python + manifest-type work; new *practice data files* need neither.
  - **The gallery is now async on both inputs.** `useSaintsCatalog` previously rendered the bespoke cards synchronously (imported array) and only the Pictorial Lives index warmed in. Both blobs now warm from Hearth, so `build()` tolerates either being `undefined` and the wall fills in as bytes land — consistent with how the index already behaved. Saints were never part of the offline starter pack, so nothing regressed there.
  - **`id` is the image stem.** `cardImage` is derived as `{ uri: hearthAssetUrl('saints/${id}.webp') }` in the catalog — the data file carries no image URL, so a card's id must match its `content/saints/<id>.webp` filename (snake_case, as before).
  - **The per-saint i18n blocks were deleted, not kept as fallback.** Leaving 66 dead `saints.<id>` entries would have invited the next author to "also" edit them. The `saints.*` UI strings (title, group, encounter, etc.) stay in i18n; only the per-saint name/feast/patron/prayer left. The old `feastDay` string ("October 1") was dropped entirely — it was only ever a fallback for cards without a feast, and every bespoke card has one, so `feastLabel` is always computed from `feast` via `Intl.DateTimeFormat`.

- **Opus Dei daily content — feed + Gospel commentary tab + meditation practice (2026-06-16).** Three opusdei.org surfaces, all scraped at runtime by `apps/app/src/sources/opus-dei/` (mirroring the Vatican News source): a "From Opus Dei" row on Explore (Atom feed), a "Reflection" tab inside `practice/gospel-of-the-day` (between Gospel and Word of the Pope), and a new `practice/opus-dei-meditation` surfaced as an Explore featured-carousel block. Gotchas that bit:
  - **The `lastarticles.xml` feed is bare Atom with no images** (only `xmlns="…/Atom"`; no `media:`/`enclosure`). Card art comes from each article page's `og:image` in a second per-article fetch — the same two-step the Vatican row uses for thumbnails. The CDN image URL takes a `&w=` width param, so we shrink the 1200px social image to 600 for cards.
  - **Article body lives in `<div class="imperavi-body" itemprop="articleBody">`.** Scope all lookups to `<main>` first — the page header carries its own hidden `<h1>Opus Dei` (logo) and sidebars carry "related" modules that would otherwise leak in.
  - **Gospel page = passage + `<hr>` + commentary.** EN uses a literal `<hr>`; both langs head the reflection with a `Commentary`/`Comentário` `<p><strong>` label. We keep only the reflection (the scripture is already the practice's Gospel tab). PT pages don't always have a parseable `<hr>` form, so prefer the label paragraph and fall back to `<hr>`.
  - **Per-run `collapseWs().trim()` mashes inline-citation spacing.** The Vatican parser trims each text node; that's safe for its plain `<p>` markup but on Opus Dei (heavy inline `<em>` scripture + `<a>` cross-refs) it turned `behaviour, <em>the</em>` into `behaviour,the` and `(1 <em>Kings</em> 20:43)` into `(1Kings20:43)`. Fix: collapse but **don't trim** text nodes, then trim only the paragraph's first/last run. A regex assertion (`not.toMatch(/\w,\w/)`) guards it.
  - **Footnotes have two forms.** In-body refs are `<a id="_ftnref1"><sup>[1]</sup></a>` (the `[1]` is stripped as a footnote-ref run); the trailing definition list is separate `<p>`s opened by `<a id="_ftnN">`. The `[n]` marker gets eaten before any text check, so detect definition paragraphs **structurally** (first child `<a id="_ftnN">`), not by leading-`[1]` text.
  - **Native-only.** Pages + feed are CORS-blocked, so the producers/row degrade to a link-out on web and on any failure; `dateScoped: true` keys the per-day cache. The producers' attribution/link-out emit a new **`link` primitive** (`{ type: 'link', text, href }`, rendered by `components/prayer/LinkBlock.tsx`) — added so sources stop embedding raw URLs as prose and external links are tappable (in-app browser native / new tab web), with the open behavior owned by the renderer. Validated against real (trimmed) page markup in `sources/opus-dei/__fixtures__/` via `parse.test.ts` — eyeballing rendered markdown (WebFetch) hid the spacing/footnote bugs; only running the raw HTML through the parser surfaced them.

- **EF display calendar unified onto the Divinum Officium engine (2026-06-16).** The EF half of the calendar (card, month grid, day detail) was the last consumer of the curated `content/liturgical/entries.json` (via `buildYearCalendar`), so it could drift from what the EF Mass/Office actually celebrates. New `buildDoYear` (`@ember/divinum-officium`, `kalendar/year.ts`) loops `resolveDay` — the *same* precedence/occurrence authority the EF Mass uses — across the year and emits a neutral row shape; the app's `buildDoYearCalendar` maps it onto the `Map<string, DayCalendar>` every consumer already expects (mirrors `buildOfYearCalendar`). Branch in `useYearCalendar`: OF→`buildOfYearCalendar`, EF→`buildDoYearCalendar`. Gotchas:
  - **The winner's name is already in `state` after `occurrence` — no section loading needed.** `resolveDay` returns `{ …, state }`; `state.srank[0]`/`state.trank[0]` hold the celebration's Latin name (from the Kalendarium), and `state.sanctoraloffice` says which cycle won. So the calendar never needs the (expensive) officestring text assembly. Added `sections?: boolean` to `resolveDay`: when `false` it returns right after the precedence/duplex block with empty section trees. Also added an optional `directorium?` param so the year loop builds the transfer/version tables **once** instead of 365×. Both are behavior-preserving for existing callers (defaults unchanged; the differential gate still passes).
  - **DO names are Latin, full stop.** The file `Rank` line's *name* field is usually empty (e.g. `Sancti/01-25` = `;;Duplex majus;;4;;…`); the human name comes from the Kalendarium table, which is **Latin only**. So "full-canonical" EF = Latin sanctoral titles (fitting for the EF) and **no descriptions** (DO has no prose). Temporal names are left empty so `useCelebrationDisplay` resolves them via `getLiturgicalDayName` (vernacular) — exactly the OF temporal pattern. `localizeContent` falls back to the first available value, so a `{ la }`-only name renders. The hybrid (curated names over DO precedence) is still possible later without rearchitecting.
  - **Rank scale (Rubrics 1960), calibrated empirically against real data:** I/II-class are 6–7 / 5, III-class the doubles (3–4), IV-class lesser feasts, simplices/commemorations below ~1.5. `normalizeEfRank` maps the DO number (+ rank-text for vigils) to `RankEF`. Don't trust the rank *text* alone — the shared files still carry pre-1960 class words ("Duplex majus") under the 1960 version, so the *number* is the reliable signal.
  - **Surfacing rule:** sanctoral winner → show it; else privileged temporal (rank ≥ 6: Advent/Lent/Passion/Easter Sundays + Lord's feasts) → show it; else a commemorated saint (srank present) → show it so summer ferias still read as a saints' calendar; else skip (ordinary green Sundays rank 5 / plain ferias). Mirrors the OF "drop Ordinary Time Sundays" behavior.
  - **Perf:** a full year resolves in ~0.85s with `sections:false` + shared directorium; query is `staleTime: Infinity`. First cold build still fetches the per-day tempora/saint blobs (the same the EF Mass would), so it's heavier than the OF's two static JSONs — acceptable for a forever-cached query; precompute is a possible later optimization.
  - **`@ember/divinum-officium` stays pure** (no `@ember/liturgical` dep): `buildDoYear` returns a DO-native row (`DoCalendarDay`) and the *app* owns the `RankEF`/`DayCalendar` mapping, so the package keeps its in-package fs-loader tests (`year.test.ts`) with no curated-data coupling.
  - **Env wrinkle:** `@ember/divinum-officium` is a declared app dep but its `node_modules/@ember/divinum-officium` symlink was missing in the fresh container (stale install), so `tsc`/Metro couldn't resolve it (pre-existing — it broke `do-mass.ts` too). `pnpm install` (or recreating the symlink) fixes it.

- **Celebration-of-the-day / calendar unified onto the canonical OF authority (2026-06-14).** The home celebration card, month grid, and day detail used to read a *separate* hand-curated `content/liturgical/entries.json` via `buildYearCalendar`, while the Mass read the canonical MR statics via `resolveOfDay`. They drifted: `entries.json`'s Sacred Heart `anchor_relative` rule resolved to **Easter+61** (Friday after Trinity) instead of the correct **Easter+68**, so the solemnity showed in the Mass but vanished from the card. Fix: a new `buildOfYearCalendar` (`@ember/mass`, `of/calendar/year.ts`) loops `resolveOfDay` across the year and emits the **same** `Map<string, DayCalendar>` shape `buildYearCalendar` does, so no consumer changed. The hooks branch OF→new builder, EF→entries.json (EF kept as-is). Gotchas worth remembering:
  - **Rank separator mismatch.** `@ember/missal-schema` `Rank` is hyphenated (`optional-memorial`); `@ember/liturgical` `RankOF`, the `calendar.rank.*` i18n keys, and `rankColors` use underscores (`optional_memorial`). The builder must `normalizeRank` or badges/colors silently break.
  - **Temporal celebrations carry no title in the *statics* — the name lives in the Mass formulary.** `resolveOfDay` returns a celebration for *every* day (ferias, ordinary Sundays) and `temporal.json` entries have no `title` (only the *formulary* does, on all 390 temporal formularies). So the builder leaves temporal `entry.name` empty and the UI resolves the title from the formulary via `useCelebrationDisplay` (the single source of truth for both title and description), falling back to `getLiturgicalDayName` while it loads / for the month-grid a11y labels + the upcoming banner (which don't fetch formularies). `temporal-notability.ts` is *only* the filter — which temporal days are surfaced (the solemnities/feasts of the Lord by `tempore.*` ref or `getOfLiturgicalPosition().specialDay`; ordinary Sundays/ferias dropped). Christmas's four Masses collapse to one display entry. (Earlier this baked bilingual title strings; that duplicated the formulary titles, so it was removed — names now have one home.)
  - **Descriptions live in the formulary, not the calendar.** `of-data`/the calendar statics carry *no* description — the "Sobre esta celebração" prose is `MassFormulary.description` (`content/of/formularies/**`). The card/detail load it on demand via `useFormularyDescription(ref)`. Coverage is lopsided (**207 pt-BR vs 2 en-US**); we deliberately do **not** fall back to the Pictorial Lives saint reflection (it isn't liturgical) — when the formulary has no description the card/screen simply omit it.
  - **HDO needed a new single source.** The 11 holy-day-of-obligation flags lived only in `entries.json`; once OF stopped reading it, `of/calendar/hdo.ts` became the canonical OF HDO table (`obligations.ts` still reads `principal.entry.holyDayOfObligation`, now set by the builder).
  - **`saintArtMap` was removed.** It bridged calendar ids → the 14 curated holy cards to show art on the celebration card, but it only fired on ~12 feasts/year and duplicated the dedicated Saint-of-the-Day card's Pictorial Lives portrait. `useSaintOfDay` now returns just `{ celebration }`; the celebration card/screen render typographic-on-tone (no holy-card image). (En route it was briefly re-keyed to formulary refs before being dropped.)
  - **OF scope follows content language** (`scopeForContentLang(contentLanguage)`), matching the Mass — *not* the `jurisdiction` pref the EF override path uses.
  - Retired the now-dead generated `content/liturgical/of-calendar.json` + `scripts/build-of-calendar.mjs` (only a stale comment + one test consumed it).

- **Intermittent app-wide touch/scroll freezes = blocked JS thread; the three culprits and the new guardrails (2026-06-12).** Symptom signature: touches and scroll dead but the bottom tab bar still responds — the tab bar is UIKit-native (expo-router `NativeTabs`), everything else needs JS. Diagnosed three independent blockers; all fixed, plus a dev-only stall detector (`lib/stallMonitor.ts`, mounted in `_layout.tsx`) that logs `[stall] JS thread blocked ~Xms` when its heartbeat gap exceeds 500 ms.
  - **expo-file-system's new API is half synchronous.** `bytes()`/`text()`/`base64()` are async, but `write()`, `delete()`, `create()`, `list()`, and the `exists`/`size`/`modificationTime` getters are **synchronous JSI calls** that block the JS thread. Never use them in hot paths or loops: the old `listCachedBlobs` triple-nested `list()` walk was thousands of sync stats for a near-200MB cache (now yields per shard dir), and every downloaded blob used to be `write()`n on the JS thread. For downloads, `File.downloadFileAsync(url, file, { idempotent: true })` is disk-to-disk native (verified in `FileSystemModule.swift`: rejects on non-2xx, network error, and existing destination without `idempotent`) — bytes never enter JS. `store.ts:ensureBlobCached` stages into `blobs-tmp/` and atomically `move()`s into `blobs/` after validation, so a killed app can't leave a truncated blob behind a forever-trusted hash.
  - **`catalogVersion` bump-storms re-rendered the app 3× per launch for nothing.** The post-boot background refresh re-fetches `catalog.json` and re-warms manifests every launch; `setCatalog` used to unconditionally invalidate + bump (one app-wide re-render of every `useCatalogVersion` subscriber), and both warm functions bumped again even with zero new manifests. Now `setCatalog` skips when `generated` matches (non-empty guard keeps test semantics) and the warm functions only notify when they actually loaded something — a no-change refresh causes zero re-renders. Also, the eviction pass (`pinnedHashes()` manifest-tree walk + the sync stat walk) now runs at most once per 24h via `content/cacheMaintenance.ts`. Pattern to keep: anything that "refreshes" on a timer/launch must prove change before notifying.
  - **Practice screens mounted their whole primitive tree synchronously.** Mass/offices render hundreds of nested Tamagui blocks inside a plain ScrollView; mounting them all at navigation time blocked JS through the whole transition. `hooks/useProgressiveCount.ts` chunks the mount (16 first, +24 per post-interaction frame, reset on sections-identity change); applied in `PracticeFlowView` and the chapter reader. Limitation: top-level only — a flow that is one giant `container` doesn't benefit. The book reader is a WebView and needed nothing.
  - Also: `expo-file-system` is now statically imported in `store.ts`/`mediaDownload.ts` (Platform-guarded) instead of a conditional `require` — the package ships a web variant, and bare `require` bypasses vitest's `vi.mock`, making the native path untestable (`store.native.test.ts` forces `Platform.OS='ios'` with a stateful FS mock).

- **OF Mass woven ordinary + the cache/dev-server traps that hid it (2026-06-11).** Rebuilt the OF Mass into `producer/mass-of` (direct-to-primitives, an Ordo spine weaving Order + propers, chip selectors). The model/extractor details live in `docs/mass-rework-journal.md`; the *gotchas* that cost the most time:
  - **A producer's output cache has no TTL and ignores the corpus hash.** `runCachedSource.ts` keys SQLite on `producerVersion` + prefs only. The mass producers read corpus blobs whose hashes change every build, so a static `version: '1'` renders the **old corpus forever** after a rebuild — the app fetched the new `catalog.json` but never the new order blob, and since `_site/blobs/` *accumulates* old blobs (content-addressed, never pruned) the stale catalog's blobs still 200'd with no error. Fix: `get version() { return \`2:${getCatalog().generated}\` }` so the catalog's build stamp busts the cache. Any corpus-reading producer must do the same.
  - **`expo start --web` does not serve native bundles.** Testing on iOS while only the `--web` dev server ran → `index.bundle?platform=ios` is a hard 404, so the device silently kept an old bundle and `fetchHearth` fell back to **production** Hearth (old published corpus) on local-timeout. "As if you did nothing" almost always = the device isn't loading this worktree's JS/corpus, not a code bug. Use `expo start --dev-client` (= `pnpm start`) for native; check the Hearth + Metro access logs for the device IP before suspecting code.
  - **`SelectBlock` only lazily preprocesses non-default branches from `rawSections`.** A `select` whose options carry pre-built `children` (no `rawSections`) rendered **blank** for every option except the default — `SelectBranch` returned `undefined`. Producers that emit final primitives into select options need `SelectBranch` to render `children` directly (fixed).
  - **Drop-caps: the upstream `<span class="cap">` lost the boundary space in the parser**, gluing `O`+`Senhor`→`OSenhor`, `A`+`graça`→`Agraça`. The space is recoverable: `segments-to-lines.ts` keeps the following text node's leading space; `richtext.ts` (baseline already trimmed) uses next-char case (glue a lowercase continuation, space before a new capital). Drop-caps are now eliminated corpus-wide.
  - **`''.join` ≠ the renderer.** Auditing corpus text with Python `''.join(seg.text)` shows `comSenhor` where the renderer's `joinLine` adds the inter-segment space (`com Senhor`). Don't flag "missing space" scannos from a no-separator dump.

- **`select on: "hour"` silently resolved to hour 0 in every practice (2026-06-11).** `useToday()` deliberately returns the *logical day at midnight* (4am-cutoff rule; its doc warns against calling `.getHours()`), but `usePracticeContent` passed it as `FlowContext.date`, and the engine's `hour`/`timeOfDay` context values read `date.getHours()` — so hour-mapped selects (liturgy-of-the-hours, divine-office) always saw hour 0 and defaulted to the night option. Fix at the engine boundary: `FlowContext` gained an optional wall-clock `now` that `hour`/`timeOfDay` prefer, and the practice-content query passes `new Date()` plus the current hour in its query key so the default follows the clock through the day. Any future host that resolves hour-mapped flows must pass `now` — `date` alone reads as midnight.

- **iBreviary LOTH producer — session-cookie date setting, two markup dialects, ora_media splitting (2026-06-10).** `producer/breviary-of-the-day` (`apps/app/src/sources/ibreviary/`) scrapes ibreviary.com/m2 for the `liturgy-of-the-hours` practice. The non-obvious mechanics:
  - **The date can only be set by POST.** `GET /m2/opzioni.php?giorno=…` silently ignores the params; only a form-urlencoded POST (`lang`, `giorno`, `mese`, `anno`, `ok=ok`) updates the `PHPSESSID` session, and `GET /m2/breviario.php?s=<section>` then serves that day. RN's `response.headers.get('set-cookie')` is unreliable, so we never parse cookies — the native HTTP stacks persist the jar themselves. The trade-off: the session is *shared mutable state*, and `SelectBlock` prefetches all hour branches at mount, so concurrent fetches would race the date. `session.ts` serializes every POST+GET pair through a module-level promise chain. **All iBreviary traffic must go through `fetchSectionHtml`** — any fetch outside the queue reintroduces the race.
  - **The two editions speak different HTML.** PT marks up one `<p>` per stanza/rubric (`p.rubrica`, centered `<strong><span class="rubrica">` psalm titles, explicit `V.`/`R.` spans); EN packs whole sections into giant `<p>`s with `<br><br>` separators, `span.capolettera_piccolo` section labels, and `—`-marked responses. The parser therefore works on a run/line/block model (runs styled by enclosing span class, lines split on `<br>`, blocks on blank lines or `</p>`) instead of per-paragraph classification. `******` is the reliable junk boundary — donate/newsletter links and the alternate-texts appendix always follow it.
  - **Terce/Sext/None are one page.** `s=terza|sesta|nona` return empty shells; the canonical little hours only exist on the combined `s=ora_media` page (common psalmody + one block per hour + shared ending). `splitDaytime` recomposes each hour; PT repeats the oration per hour while EN prints the concluding prayer once in the last segment, so the EN conclusion slice starts at the `CONCLUDING PRAYER` rubric.
  - **`lang=pt` is the European Portuguese LOTH** (pt-PT, "Vós" forms) — accepted for pt-BR users since Brazil's CNBB edition isn't available. `lang=la` (Liturgia Horarum) ships as a primary edition for `contentLanguage: la`; `lang=vt` (Vetus Ordo) remains future work — the lang/hour tables in `config.ts` are the only place to extend.

- **Select tabs switch client-side now — no whole-flow rebuild (2026-05-31).** Tapping a tab in a `select` block used to flash the full-screen "Oremus" threshold: `selectOverrides` was part of `usePracticeContent`'s React-Query key, so any tab change re-resolved the *entire* practice flow, dropped `data` to `undefined`, and `derivePracticeFlowStatus` fell to the loading branch. The engine had deliberately materialized **only the selected** option's body (`resolve.ts` static + `select.ts` from-data, "for responsiveness"), forcing that re-resolve to fetch the other branch. Fix is **lazy + prefetch**, hinging on one insight: engine resolution (`resolveSection`) is synchronous and cheap — it emits a `RenderedSection` tree still holding unresolved `include`/`reading` markers; the **expensive network work is in `preprocessFlow`** (resolving those markers via ContentSources). So: (1) the engine now materializes **every** branch's *structure* in its single walk, each with its own `templateVars`/`flowData[as]` binding — cheap, no network; (2) `preprocessFlow`'s `select` case preprocesses **only the selected branch** into `children` (so the default paints with the flow) and carries the raw `RenderedSection[]` for all options on a new `ContainerOption.rawSections`; (3) `selectOverrides` is **removed from the main query key** (`usePracticeContent` resolves with `{}` = engine's auto/default), so a tab change never touches the main query; (4) the renderer switches client-side — `SelectBlock` holds local `activeId`, and `SelectBranch` either renders the default's `children` or runs a per-branch `useQuery(['select-branch', practiceId, overrideKey, optionId, lang, translation, todayKey], …)` that preprocesses `rawSections`, showing a `Skeleton` only while that one branch resolves. `SelectBlock` `prefetchQuery`s the non-default branches on mount, so a tap is usually instant. The branch runtime (`queryClient`/prefs/date/programDay) reaches `SelectBranch` via a `PreprocessProvider` context wrapped around `PracticeReady`'s body — the one sanctioned exception to "blocks never fetch", localized to the inherently on-demand `select`. **Completion gotcha:** `findTrackIds`/`walkRendered` (extracted to `PracticeFlow/findTrackIds.ts`) walk *all* select options; with every branch now materialized that would advance reading cursors for branches never prayed, so they now descend only the effective `selectOverrides[overrideKey] ?? selectedId`. **Known limitation (acceptable):** because switching is client-side and the main flow resolved with the default pick, sections *outside* a select that template off its choice (`as: view` referenced elsewhere) won't live-update on a tab change — no current content does this; selects are self-contained pickers.

- **PDF book ingestion: `My Catholic Faith` (Bp. Morrow, 1949) end-to-end pipeline (2026-05-18).** First book imported directly from a PDF source rather than transcribed HTML/text. The source is a 423-page two-column pictorial catechism with 193 numbered lessons. Pipeline lives at `scripts/import-my-catholic-faith.py` and is fully re-runnable: it downloads the PDF to `.cache/`, wipes `content/books/morrow-my-catholic-faith/{en-US,images}/`, and rebuilds everything from page bounds detected in the PDF itself. Validation: `scripts/validate-my-catholic-faith.py`.

  **Layout-aware extraction beats `pdftotext`.** The PDF has 11pt body text, 9pt commentary expansions, 11pt-bold questions, italic image captions, and a 12pt+ bold lesson heading per chapter. Plain `pdftotext` flattens the 2-column layout in physical order (left-then-right within each line band → reads wrong). The pipeline uses pymupdf's `get_text("dict")`, splits each line into runs of consecutive spans sharing `(size, bold, italic)`, classifies via font reference (`Type3 (428 0 R)` = bold, `Type3 (598 0 R)` = italic, `Type3 (504 0 R)` = regular — flags=0 on every span, useless), then sorts left/right columns by `(int(y0 // 7), x0)`. **Floor division, not round, is critical:** `round(y0/5)` for adjacent fragments at y=522.515 vs y=522.488 put them in *different* buckets (105 vs 104) because they straddle the .5 rounding boundary — that's what was scrambling bold-question and regular-answer fragments across many lessons. Column split is at x=247 (left col tails end ~245, right col starts cluster at 250+; gutter is sparse at 246-249), not 250 — that fix unscrambled the 2-up Q&A layout in lessons 80/82/130.

  **Whitespace artifacts are baked into the PDF text stream**, not extraction-method artifacts — `pdftotext`, pymupdf's `dict` mode, and `words` mode all emit `ofthorns`, `ifwe`, `endofthe`, `BlessedTrinity`. Fix: `split_glued(token)` with three paths gated on confidence. (1) CamelCase boundary `[a-z][A-Z]` + nested wordninja per piece; (2) ALL-CAPS path that lowercases, runs wordninja, re-uppercases, handles trailing `'S`; (3) general wordninja with at-least-one-glue-word requirement; (4) binary fallback for tokens with zipf≈0 (recovers `ofsin` that wordninja over-splits to `of s in`). Each path validates pieces via `wordfreq.zipf_frequency` (≥3.0, or 2.5/2.7 in the camel+glue and all-caps paths). Tested against 25 cases including proper nouns (`LaRavoire`, `Krishnagar`, `DeVico`) and rare ecclesiastical terms (`Indefectibility`, `Sacramentals`, `Manichaean`) — none mis-split. Closed-class glue whitelist + a small `CATHOLIC_VOCAB` (saints, places, biblical verbs) covers the residual long-tail.

  **Section dividers are not chapters.** The printed TOC italicizes ~10 sub-section labels like "APOSTOLICITY OF CATHOLIC DOCTRINES" between numbered lessons — visually they look like headings (12pt bold), but they aren't separate chapters. Two consequences: (1) lessons including their following divider can appear as 3-page rather than 2-page spans, which is correct — the divider stays with the previous chapter per user direction; (2) when a divider appears at the top of a lesson's first page (e.g. "Eminent Catholics" on lesson 65's page), `find_heading` must search past it for the actual `N. TITLE` line. Solved by tagging non-numbered 12pt-bold lines as `section` (rendered as `## H2`) instead of `heading` (`# H1`), and emitting the H1 first in markdown regardless of physical order.

  **Lesson 61's heading mixes 12pt and 13pt** in the same line ("**61.**" is 13pt, "THE POPE, BISHOPS AND COUNCILS" is 12pt). Since the fragment splitter keys on size, the two halves landed in different fragments; the heading detector saw only "61." Fix: when scanning for the heading line, merge adjacent same-line bold≥12pt fragments before pattern-matching.

  **Running-header band y-coordinate is razor-thin.** Page numbers + "MY CATHOLIC FAITH" / running title sit at y≈18; lesson 66's heading sits at y≈24. Cutoff of y<30 dropped lesson 66's heading entirely. Final rule: only drop y<30 if the line's font is *not* 12pt+ bold — a lesson heading near the top of the page is real content, not a header.

  **193/193 chapters, 189/193 header images.** Four lessons (14, 51 originally — 51 recovered, 58, 65) and one appendix have illustrations rendered as Type3 vector glyphs which `pdfimages` can't extract. Fallback: rasterize the page region above the heading via `page.get_pixmap(clip=Rect(0, 30, width, heading_y-4))` at 2× zoom and convert via ImageMagick → WebP. Recovered 3 of the 7 originally missing; the remaining 5 are genuinely imageless (the heading sits at the top of the page with no room above for an illustration).

  **Auto-generated TOC mirrors source.** `book.json`'s TOC titles are regenerated on each import run from the extracted lesson headings, so the manifest can't drift from chapter content (Judgement/Judgment, Neighbour/Neighbor, Colors/Colours — the PDF uses British spelling intermittently; trying to hand-type a "canonical" TOC produces 21 mismatches against the actual H1s).

  **Lesson 193 + appendix split.** The PDF runs the final lesson + Church Year + Most Important Prayers + Index as one continuous text from page 406 to end. `APPENDIX_OVERRIDE` clamps lesson 193 to pp 406-407; `APPENDICES` carves Church Year (408-411) and Most Important Prayers (412-415) into their own chapter files; the index (p416+) is dropped — it's a printed-book navigation aid that doesn't translate.

  **Sub-agent audit loop.** After the initial import, two rounds of sonnet sub-agents (7 + 4 parallel) reviewed chapter batches and surfaced fixable artifact classes beyond the initial ones: glued-on-2-char artifacts (`ofa`, `Iam`), pronoun-glued verbs (`Himselfgave`/`Himselftaught`), space-split words (`Via ti cum`, `dir i ment`, `A men` → `Amen`), bold-question mid-sentence with `?` typeset in regular weight, line-break hyphens not re-joined (`im- pediments`), numeric glue (`of350`), and the `Chruch` source typo. Each got a targeted fix; the regex-driven `SPLIT_FIXES` table feeds into a `PROTECTED_REJOINED` set that prevents `split_glued` from immediately undoing the rejoin (e.g. wordninja sees `Amen` and wants to re-split to `A men`; the guard short-circuits). The `clean_text` second pass runs on the rendered markdown (skipping image-ref lines) to catch typos that only manifest after title-casing converts the all-caps source heading.

  *Locked in:* (a) **Producer** = built-in code function in its own package, registered by id, returning either `FlowBlock[]` (flow consumers) or cleaned HTML + anchor sidecar (reader consumers) — no forced unification.   - **One Movement projection, two event streams.** Petitions and thanksgivings share `Map<id, Movement>` discriminated by `kind`, but their events are distinct (`IntentionRaised`/`Answered`/`Expired`/`Retired` vs `ThanksgivingOffered`/`Updated`/`Retired`). Lives in `apps/app/src/content/engineContext.ts`.
  - **Engine doesn't know about clocks.** `ec.windowFor(level, forward)` takes only level + direction; the host closes over its anchor. There is no `ec.logicalDay()` — the engine asks "what window?", the host decides what "today" means.

  - **Capture-resolution pre-fills from the projection.** `ec.resolutions.inWindow(level, starts_at)` looks up an existing resolution whose window matches the block's target window. Stops users from accidentally creating duplicate resolutions across multiple Examen runs in the same day.
  - **No "Skip" controls.** Both `capture-movement` and `capture-resolution` are *optional by default*; the way to skip is to not submit. An explicit Skip button would be irreversible (once tapped, the form is gone), which is hostile when the user just wants to scroll past. The `optional` prop was removed entirely from the DSL.
  - **Explicit-open everywhere.** Capture forms (offering, capture-movement) open only on a "+ Add" tap. Auto-opening a textarea the user didn't ask for is the same anti-pattern; one consistent fix.

Useful anywhere a chrome-only section needs to gate on a single conditional child.

- **CCEL pilot conversion learnings (2026-04-28).** Ran the importer against all four pilot ThMLs end-to-end on a machine with CCEL access. Final shipped library: `ccel-classics-0.1.0.pray` (1.5MB, 656 files, 4 books / 638 chapters total). Several real-world quirks the synthetic fixture missed:

  (1) **DC fields use both element and HTML form**: CCEL ThML 1.04 uses `<DC.Creator scheme="short-form">Thomas à Kempis</DC.Creator>` (element), not the HTML `<meta name="DC.Creator" content="…"/>` form. The original extractor only looked at `<meta>` and pulled nothing. Fixed `_meta_value` to scan element form first, then HTML; added `_meta_values_with_attr` to disambiguate by `scheme`/`sub`. Author preference: `scheme="short-form"` → `scheme="file-as"` → first DC.Creator. Skip `scheme="ccel"` (the bare slug).

  (2) **`<DC.Date sub="Created">` is the digital-edition timestamp, NOT composition**. Imitation showed `1998-12-10`, Devout Life `2002-08-08`. Importer now prefers `<firstPublished>` / `<published>` from `<generalInfo>`; only accepts `DC.Date` if it's a pre-1900 4-digit year. For modern translations / works without `firstPublished`, pass `--composed` explicitly.

  (3) **Source URL fallback**: not every CCEL file has a `DC.Identifier` URL; many only have the slug `ccel/kempis/imitation.htm`. Synthesize `https://www.ccel.org/ccel/<authorID>/<bookID>/` from `<authorID>`/`<bookID>` in `<electronicEdInfo>` when no http(s) DC.Identifier exists.

  (4) **`DC.Language` may be ISO 639-3** (`eng`, `por`, `lat`, `grc`). Extended `_LANG_MAP` to handle 3-letter codes.

  (5) **Skip CCEL navigation indexes/contents**: every CCEL file ends with a `<div1 title="Indexes">` (or `<div1 title="INDEX.">` for older files) wrapping `<div2 title="Subject Index">`, `<div2 title="Index of Scripture References">`, etc. Some also start with a print-edition `<div1 title="CONTENTS.">` table of contents. These are auto-generated reference data and produce mangled output. Filter by title (`index`, `indexes`, `contents`, `index of …`) at root level + at chapter level — covered by `_is_index_section`.

  (6) **CCEL chapters open with print decorations**: `<h4>The First Chapter</h4>` then `<h3>{title}</h3>` before the first paragraph. We already inject our own `# {title}` H1 from the section attribute, so any leading sub-heading inside a chapter body is redundant — strip everything until the first non-empty, non-`#` line.

  (7) **`<p> </p>` whitespace-only spacers** are common in CCEL files for visual layout. Drop them entirely instead of emitting blank "  " lines.

  (8) **Whitespace doubling around `<pb/>`**: `<p>…about\n\n<pb/>\n\nthe Trinity…</p>` produced `learnedly about  the Trinity` (double space). Added `re.sub(r" {2,}(?!\n)", " ", body)` in `render()` — collapses 2+ spaces to 1 except when they form a Markdown hard break (`  \n`).

  (9) **`<verse>` in Augustine's Confessions** is a poetic line group (children: `<l>`) — same as `<lg>`. Added to the `lg/verse` handler.

  (10) **Recursion bug in unknown-block fallback**: `<table>` was in `_BLOCK_TAGS` but had no handler in `_walk_block`, so it fell through to a `_walk_inline(el, b)` call that routed back to `_walk_block` based on `_BLOCK_TAGS` membership → infinite recursion. Fixed by walking children directly in the fallback rather than re-dispatching the same element.

  (11) **CCEL `<div1 type="Book" title="Preface">`**: Summa Contra Gentiles has a Preface div with `type="Book"`. Original slug logic preferred `type` over title, producing `book.md`. Re-prioritized: title takes precedence over bare type when no `n` is present.

  (12) **Long titles → filesystem errors**: Devout Life's `<div1>` titles include the full part-summary, then chapter titles add another sentence — concatenated, the slug exceeded 255 bytes (macOS filename limit). Slug now detects "PART/BOOK/CHAPTER N" prefixes and uses just the prefix (`part-i`, `chapter-vii`); falls back to the first six words of the title.

  (13) **Augustine Confessions has no `<div2>`** — the whole text is `<div1>` per book with `<p class="chapter">CHAPTER I</p>` paragraph-markers inline. `--chapter-level auto` correctly picks `div1`, producing 13 large book-length files (43–150KB each). The print-edition CHAPTER markers stay as plain paragraphs in the output; promoting them to `## H2` based on the `class="chapter"` attribute is a future refinement.

  (14) **The pilot test repo**: `timothyhartzog/theology-corpus` (small Reformed-evangelical CCEL mirror) has Aquinas's Summa Theologica and Augustine's Confessions/City of God, but most Catholic pilots had to be fetched directly from `ccel.org`. Pilot inputs live under `tmp/ccel-sources/` (gitignored).

## Architecture Decisions

- **Unified flow system replaces variants, forms, and multiple flows (2026-04-12).** The content engine accumulated four content-selection mechanisms (variants, forms, multiple flows, setKeyOverride) each built for specific practices. Decision: collapse all four into one `select` section type — a conditional branch that reads context (day of week, time of day, user preference) and picks an option. One flow per practice. Different meditation traditions become separate practices grouped by `alternativeTo`. Full spec: `docs/features/unified-flow-system.md`. Key insight: all four mechanisms were just conditional content selection with different indirection layers. The `variant` DB column was never actually written — every user was on the default.

- **DSL completeness: no practice should need custom app code (2026-04-12).** The flow JSON must be expressive enough that the app is a pure reader. The `useLiturgicalMeditation` hook (118 lines, meditacoes-ligorio only) was the clearest violation — calendar→chapter resolution hard-coded in React. Fix: `resolve` field on `FlowDefinition` with pluggable strategies (first: `"liturgical-day"`), and dynamic `prose` sections that load book chapters by template-variable chapter IDs. The resolution algorithm stays in `@ember/liturgical` but is invoked declaratively from the flow JSON, not from a practice-specific hook.

- **Visible `select` blocks require explicit app-side rendering + override wiring (2026-04-14).** After migrating practices to unified flow, the "title-only, no body" symptom appeared only on practices whose main content lived under top-level visible `select` sections (e.g., Mass). The content engine was already emitting rendered `select` sections, but `SectionBlock` had no `select` case, so those branches rendered as `null`. Fix: add `SelectBlock` UI renderer and thread `selectOverrides` state through `PracticeFlow` so selector changes re-resolve flow content live. Silent/static patterns (e.g., Way of Light, Bible+Catechism) were unaffected.

- **Large-flow `select` switching is fast only when non-selected branches are lazy (2026-04-14).** For visible `select`, eagerly resolving sections for every option on each pass caused noticeable delay on heavy flows like Mass. The UI only needs labels for all options and sections for the active option, so the engine now resolves sections only for the selected option and emits empty section arrays for non-selected options. This keeps behavior identical while removing most of the per-toggle work.

- **Unified flow completed migration path for dynamic Liguori meditations (2026-04-14).** The old `useLiturgicalMeditation` hook was removed and replaced with declarative flow resolution: `resolveFlowAsync` executes flow-level `resolve` strategies (starting with `liturgical-day`), injects template vars, preloads dynamic prose chapter text, and then renders sections without practice-specific React logic. - **Legacy slot flow/variant plumbing removed from runtime writes (2026-04-14).** Slot creation/update no longer writes a `variant` field, and the unused `changeSlotFlow` mutation path was removed from repositories/hooks. The database column remains for backward compatibility, but runtime behavior now treats slots as practice-scoped schedules with unified-flow selection handled by the engine.

This keeps the flow declarative while keeping side-effect handling inside bounded source adapters.

## Infrastructure

- **Hearth replaces the broken Expo web deploy on GitHub Pages.** The original `.github/workflows/deploy.yml` built and deployed the full Expo web app to GitHub Pages but broke after the monorepo restructure (missing `baseUrl` config). Instead of fixing it, we repurposed GitHub Pages as a static content server called Hearth. It copies Bible, propers, catechism, and saints images to `_site/hearth/v1/`, converts PNGs to WebP, and generates a `manifest.json` with SHA-256 hashes. No npm install or Metro — runs in under a minute. The `v1/` URL prefix allows future breaking changes without breaking old app versions. Base URL: `https://ember.dpgu.me/hearth/`.

- **App fetches all large content from Hearth on demand.** Bible (DRB), Catechism (CCC), EF Mass propers, and saints images are no longer bundled in the app. They're fetched from Hearth via `apps/app/src/lib/hearth.ts` (thin `fetch()` + SQLite cache wrapper) and cached in the `cache` table. This saves ~63MB from the app bundle. Source files live in `content/` at the repo root. First launch requires network — accepted trade-off. The `@ember/mass-propers` package's `PropersDataSource` interface is fully async to support this.

## Content & Licensing

- **DRB is the only viable bundled English Catholic Bible.** The Douay-Rheims (1749–1750) is fully public domain. Every other major Catholic English translation (NABRE, RSV-CE, NJB, etc.) is under active copyright. Source: `xxruyle/Bible-DouayRheims` on GitHub, MIT license.

- **Bolls.life DRB is incomplete — do NOT use it.** The Bolls.life API's Douay-Rheims translation only has 66 books — all 7 deuterocanonical books are missing. For Catholic use, always use the bundled version from xxruyle. Non-Catholic translations (66 books) work fine but need fallback to bundled DRB for deuterocanonical content.

- **ICEL copyright blocks free OF collects and antiphons.** The Ordinary Form's Collect, Entrance/Communion Antiphons, and other variable prayers are copyrighted by the International Commission on English in the Liturgy. No free API provides them in structured format. This is a hard blocker for complete OF daily Mass propers. The EF has no such problem — everything is available via Missale Meum/Divinum Officium.

- **CCC usage requires Vatican attribution.** The Catechism JSON comes from scraping vatican.va. Attribution required: "Catechism of the Catholic Church, copyright Libreria Editrice Vaticana." Vatican generally permits non-commercial educational use.

- **Divinum Officium hymns use a custom bracket format.** The source files aren't JSON — they use `[SectionName]` brackets. Needs a one-time parse step to convert to JSON. Already done for MVP hymns.

## APIs & Integrations

- **Bolls.life API: free, no auth, generous rate limits.** Base URL: `https://bolls.life`. No documented rate limits, but don't abuse it. Responses cached indefinitely in SQLite since Bible text never changes.

- **EF propers bundled from Divinum Officium, not Missale Meum API.** Missale Meum wraps DO data but only supports English + Polish. We need Portuguese too, so we parse the raw DO files directly. The bracket-based format (`[Introitus]`, `[Oratio]`, etc.) is parsed at build time by `scripts/parse-do-propers.ts` into JSON. Portuguese coverage is partial (207 vs 480 Tempora files) — the app falls back to English via `localizeContent`. Cross-references (`@Tempora/...`, `@Commune/...`) point to `missa/` or `horas/` Commune directories. Prayer macros (`$Per Dominum`) expand from `Ordo/Prayers.txt`. The Commune files for Mass sections live in `horas/{lang}/Commune/` (same files serve both Office and Mass).

- **Evangelizo: OF reading text but with quirks.** `https://feed.evangelizo.org/v2/reader.php?date={YYYYMMDD}&lang={LANG}&type=all` provides full daily reading text. Limitations: response may contain HTML that needs stripping, date range limited to ~30 days from current date, no CORS headers (may need proxy for web), and does NOT provide collects/antiphons.

- **Catholic Readings API: OF calendar + references only.** GitHub Pages hosted (free, MIT). Provides Scripture references (not full text) and liturgical day metadata. The `psalm` field sometimes references non-psalm books (e.g., Jeremiah). Cross-chapter ranges use em-dash (—). Coverage: 2025–2026 data available.

- **Universalis JSONP: potentially has OF propers but unreliable.** Claims to provide full Mass propers via JSONP. Documentation is vague and incomplete. Legacy JavaScript callback pattern doesn't fit React Native well. Worth investigating further but don't count on it.

- **Bible Gateway and API.Bible are not viable.** Bible Gateway discontinued its API and prohibits scraping. API.Bible has FUMS tracking requirements implying online-only usage, unclear Catholic translation availability.

## Technical

- **ProperSlot retries share one OF query key.** Each slot on the Mass screen (Introit, Collect, Antiphon, etc.) has its own `useProperForSlot(slot, 'of')` call, but all slots read from a single `['of-propers', dateKey, language]` query. When a user taps "Try again" on any failed slot, `ofQuery.refetch()` triggers one network request and TanStack Query re-derives every slot's section from the refreshed data — no fan-out, no per-slot request storm. Same approach for EF but each slot has its own key (EF reads from bundled/Hearth propers, independent slots).

- **Global mutation error handler uses TanStack Query's MutationCache, not per-hook wiring.** Across 39+ mutations in 10 features, none had `onError` handlers — failures were silent. Rather than retrofitting each hook, `_layout.tsx` configures `QueryClient({ mutationCache: new MutationCache({ onError }) })` once. The global handler opens `ConfirmSheet` in single-action mode with `i18n.t('error.somethingWrong')` + the error message (or `error.tryAgainLater` fallback). Per-mutation `onError` still wins: the global handler guards with `if (mutation.options.onError) return`. Trade-off: every mutation error now shows a dialog unless explicitly suppressed — acceptable because the fallback is far better than silent failure.

- **The original `getLiturgicalSeason()` was an antiphon scheduler, not a season calculator.** Its boundaries mapped to the Marian antiphon switching dates (Advent→Feb 1, Feb 2→Holy Wednesday, etc.), which don't correspond to actual liturgical seasons. The antiphon schedule is a separate traditional system with its own date ranges — decoupled from seasons in the rewrite.

- **OF and EF seasons share Easter and Advent but diverge everywhere else.** Key differences: EF has Epiphanytide (Jan 14 to Septuagesima) and Septuagesimatide (3 pre-Lent Sundays using violet), neither of which exist in OF. EF has "Time after Pentecost" where OF has "Ordinary Time." Christmas ends at Baptism of the Lord (OF, ~Jan 11) vs Octave of Epiphany (EF, Jan 13). A single `LiturgicalSeason` union with 8 values covers both forms cleanly — each form uses a subset.

- **Year-boundary edge case in season calculation.** Dec 25 is after the start of Advent, so a naive "if date >= Advent start, return advent" incorrectly classifies Christmas Day. The fix: check Christmas (Dec 25+) before checking Advent. Similarly, Jan 1 is still Christmastide in both forms.

- **Divine Office migrated to practice content format.** The custom engine (`divine-office/engine.ts`) and `PrayerFlow` component were replaced by JSON flow files + the shared content engine. Three key infrastructure changes accompanied this: (1) `practice_completions` event log replaces `practice_logs` — supports multiple completions per day and per-hour detail tracking; (2) `reading_tracks` replaces `reading_progress` — named cursors with `id` PK instead of `type` PK, enabling future track sharing between practices; (3) reading tracks are auto-advanced on practice completion whenever `tracks` are defined in the manifest. The `daily_office` and `practice_logs` tables still exist but are deprecated.

- **Unified hours/forms/flow into a single `flows` model.** Three previously separate manifest fields (`flow`, `hours`, `forms`) were replaced by a single `flows: FlowEntry[]` array. Each flow is a named, schedulable prayer sequence. Scheduling (tier, day-of-week) moved from the `Variant` type into `defaults.slots[]` in the manifest — each `SlotDefault` maps a `flowId` to a schedule. The `Variant` type was simplified: `selector`, `schedule`, and `setNames` fields were removed; the engine now receives the active flow ID as `setKeyOverride` in `FlowContext` and picks `variant.data[flowId]` directly (falls back to the first key). This made the Rosary manifest explicit: 4 flows (joyful/sorrowful/glorious/luminous) with day-of-week slots, instead of an opaque `selector: "day-of-week"` schedule inside the variant. Simple practices get a single flow with `id: "default"`.

- **Tamagui sub-theme nesting for liturgical seasons.** Wrapping the app in `<Theme name={season}>` inside `<TamaguiProvider defaultTheme={resolvedTheme}>` causes Tamagui to resolve `light` + `advent` to the registered `light_advent` theme automatically. This means any component reading `$accent` gets the seasonal color with zero changes to those components. Important: `useThemeName()` now returns the sub-theme name (e.g., `'dark_lent'`), not just `'dark'` — any code checking theme name equality must use `.startsWith('dark')` instead of `=== 'dark'`.

- **Backward-compatible wrappers for reading tracks.** `getReadingProgressByType('ot')` maps to `getReadingTrack('default-ot')` so existing consumers (settings screens, reading progress hooks) work without changes. The `ReadingTrack` type is a superset of `ReadingProgress` (adds `id` and `label`).

- **Typography is a "Ladder of Reverence", and the app is all-serif.** The old system had four font families (`$display`/`$heading`/`$body`/`$script`) each doing 2–4 jobs, so type didn't separate interface from reading from prayer. The rebuild maps the families onto seven semantic rungs (interface → rubric → apparatus → reading → prayer → label → sacred-title → peak), expressed as the `variant` prop of a single `Typography` component in `components/typography/` (see the consolidation note below). One added typeface: a medievalist **`$title`** (Junicode/IM FELL) for sacred titles. **A quiet sans (Alegreya Sans) was tried for the interface rung and removed** — it "sucked the soul out", reading as generic-app chrome against the manuscript character. So chrome stays EB Garamond and recedes via size/weight/color/case, never a sans (`variant="interface"` = quiet `$body`). The discipline that prevents a repeat of the old `$script` overuse: default to the lowest adequate rung; one rung-7 peak per screen; gold/burgundy are *sacred-only* color signals, UI stays neutral; never set raw `fontFamily`/`fontSize`. See `docs/design/design-system.md` § Typography and `docs/CONVENTIONS.md` § Typography.

- **Junicode is not on Google Fonts — bundle the OFL TTFs locally.** IM FELL English ships as an `@expo-google-fonts/*` package, but Junicode has no such package; the full normal-width family (Light→Bold + italics, from the `psb1558/Junicode-font` v2.224 release) is committed under `apps/app/assets/fonts/` and loaded via `require(...)` in `_layout.tsx`, same as `UnifrakturMaguntia`. (Historical: the two medievalist faces once both loaded every boot behind `flags.sacredTitleFace` for live A/B — see the resolution below.)

- **`$title` A/B resolved: Junicode wins; IM FELL English removed entirely.** After comparing on real screens, Junicode was kept — warmer, and it carries the full weight range (IM FELL ships only 400 + italic, so every other weight flattened). Dropped the `@expo-google-fonts/im-fell-english` dependency, the `sacredTitleFace` flag, and the dual-wiring in `fonts.ts`; `$title` now points straight at Junicode. The A/B-via-flag pattern was cheap and worth repeating for future face decisions.

- **The Ladder is one `Typography` component with a `variant` prop, not N intent-named components.** The first build gave each rung its own export (`UiText`, `Rubric`, `SacredTitle`, `SectionLabel`, `CeremonialMark`, …). That was reconsidered: the rungs are *roles* but their differences are mostly font/size/color/case, which is exactly what a Tamagui `variants` block expresses — so the whole ladder collapsed into one `styled(Text)` whose `variant` is the rung (`interface` default, `rubric`, `annotation`/`reference`/`verse-number`/`caption`/`whisper`, `label`/`marker`, `sacred-title`, `ceremonial`/`drop-cap`, plus `screen-title`). One orthogonal `tone` variant (`default`/`muted`); everything else is a pass-through `Text` prop. The apparatus rung is where the old N-component approach was most bloated (5 near-identical exports). Two deliberate carve-outs stay separate: **`PrayerText`/`PrayerLines`** (rungs 3–4) because they call `useReadingStyle` — a styled component can't run hooks — and the thin prayer-domain wrappers `SectionHeading`/`SectionMarker`, which now just render `<Typography variant="label|marker">`. Gotcha re-confirmed: `styled(Text)` infers the `fontSize` type from the default `$body` font (tokens 1–5), so hero/peak sizes inside variants must be **numeric** (`fontSize: 48`/`46`), and call sites needing `$6`/`$7` (oversized `$title`/`$display` tokens) keep a `'$6' as any` cast.

- **Reading leading must be a ratio, not absolute px.** `useReadingStyle`'s line-height was an absolute-px array indexed independently of font size, so at large font sizes it collapsed toward ~1.1 (cramped). It now selects a *ratio* (`readingScale.leadingRatio`, default ≈ 1.5) applied as `round(fontSize × ratio)`, so leading scales with size. `preferencesStore` steps stay 1–7 (no migration; the persisted integer just selects a ratio bucket). The book-reader WebView already derived a unitless ratio (`bookReader.ts buildConfigCss`), so only the native path changed. `useReadingMaxWidth()` adds a ~34em measure cap consumed by reading *containers* (not per `<Text>`, which would fight flex / bilingual side-by-side).

- **RN `<Text>` won't switch face on inherited weight/style.** With `fontFamily` pinned, a nested `fontWeight="700"` doesn't pick the bold face — you must resolve the concrete face name from the font's `face` map (see `ProseBlock` `emphasisStyle`). Only EB Garamond ships bold/italic faces; the other user-selectable reading fonts load Regular only, so non-EB fonts fall back to synthetic weight/style. Also: a `styled(Text)`'s `fontSize` token type is inferred from the *default* `$body` font (tokens $1–$5) regardless of its own `fontFamily`, so `$6`/`$7` sizes on `$title`/`$display` components need a numeric value or a cast.

- **Rubrics restyled from section headers to Missal-style italic.** The `RubricLabel` component was originally uppercase, letter-spaced Cinzel text — functioning as a section header. Real Missal rubrics are italic, sentence case, in red (hence "rubric" from Latin *rubrica*). The component was restyled to italic body font with accent color. All 40+ flows that used rubrics as section labels ("Opening Verse", "Hymn", "Psalmody") were migrated to `subheading` type instead. Mass flows received comprehensive rubrics at every major gesture (genuflections, bows, signs of the cross, hand positions) following the 1962 Missale Romanum (EF) and GIRM (OF).

- **Mass view modes use grouped flows, not engine filtering.** We considered two approaches for Propers/Readings views: (1) a single flow with engine-level section filtering by `viewMode` in `FlowContext`, or (2) separate flow JSON files with a `group` field on `FlowEntry` linking sub-flows to their parent. We chose (2) because it's zero engine complexity — just JSON content + a `group` field. The `ViewModeSelector` component renders a segmented control inside the pray screen; switching modes swaps the loaded flow via `useState` without navigation. The `FlowButtons` component on the detail screen filters out grouped flows so only primary flows appear as entry points.

- **OF propers: Liturgia Diária API gives complete PT-BR propers, Evangelizo gives EN readings only.** The `liturgia.up.railway.app/v2/` API (from `Dancrf/liturgia-diaria` on GitHub) provides all OF Mass propers in Portuguese — collects, prayers, antiphons, and readings with full text. Found via `servusdei2018/awesome-catholic`. For English, only readings are available (Evangelizo) — ICEL copyright blocks English collects/antiphons. Universalis was investigated thoroughly and confirmed to only serve readings despite vague docs. See `docs/content/of-propers-research.md` for full research.

- **Evangelizo has no psalm response/refrain.** The API returns only the psalm verses — the responsorial refrain (e.g., "Keep me, O God, for in you I take refuge") is not included. The Liturgia Diária API provides this for PT-BR via the `refrao` field, but no structured English source has been found. This is a known English data gap.

- **Evangelizo response format is HTML, not JSON.** The `feed.evangelizo.org` API returns HTML with `<br />` tags, `&quot;` entities, and `<font>` tags for citations. Sections are separated by triple `<br />` tags. Needs HTML stripping and section detection based on content (Psalms → psalm slot, Gospel → gospel slot, etc.).

- **Proper form must come from flow JSON, not user preferences.** A user may prefer the EF liturgical calendar but still be viewing the OF Mass practice. The `form` field on `proper` sections in flow JSON determines which data source is used. This is passed through the content engine to the `ProperSlot` component.

- **EF Mass propers: Tempora/Sancti precedence must use the liturgical calendar.** A naive merge (Sancti overlays Tempora) is wrong — on Holy Thursday 2026, St. Francis of Paola (Sancti `04-02`, III class) was overriding the Triduum epistle with an empty `Lectio`. The fix: use `DayCalendar.principal` from `buildYearCalendar()` to determine which source provides the Mass. If the principal's `category` is temporal (`solemnity_temporal`, `feast_of_the_lord`, `liturgical_season`) → use Tempora propers. If it's a saint category → use Sancti propers. Never merge both. Fallback without a calendar: Tempora-only (safe default — missing a saint is less wrong than overriding the Triduum). Future: commemorated saints should contribute only their Collect (Oratio) to the Mass of the day.

- **Friday abstinence rules differ significantly by jurisdiction.** Universal canon law (Canon 1251) requires meat abstinence on ALL Fridays year-round. However, the USCCB (1966 Pastoral Statement) replaced obligatory Friday abstinence outside Lent with a general penance obligation (any form). The CNBB (Brazil) kept year-round Friday abstinence but allows substitution with charity or other food — subtly different from the US approach. EF (1962 rubrics) maintains complete abstinence on all Fridays plus vigil days. Modeled as `AbstinenceLevel: 'full' | 'partial' | 'penance-required' | 'none'` in `packages/liturgical/src/obligations.ts`.

- **Books are raw HTML/markdown chapters inside `.pray`, not EPUB files.** The original plan was to embed `.epub` files in `.pray` archives and use epub.js or Readium for rendering. This was simplified: book chapters are raw `.html` or `.md` files bundled directly in the `.pray` zip under `books/{bookId}/{lang}/`. The app reads them from disk via `expo-file-system` and renders in a WebView with CSS column pagination. Markdown files (`.md`) are converted at runtime using `marked` + `marked-footnote` (~45KB). This eliminated the pandoc build dependency, JSZip from the reader path, and the double-unzip problem (.pray ZIP containing .epub ZIP). Content structure: `content/libraries/{libraryId}/books/{bookId}/{lang}/`.

- **Ember days in calendar-data.ts are placeholders, not properly resolved.** The `ember-days` entry has a dummy fixed date (Jan 1) and `ember-days-advent` only resolves to the Advent Ember Wednesday. - **Foliate paginator has no public `index` getter — and `setStyles` DOES exist (2026-06-11, post-#266 fixes).** Two long-standing wrong assumptions in `bootstrap.raw.js` caused the reader to jump to the start of the book on *every* settings change (font, palette, flow toggle, rotation): `setConfig` restored position via `paginator.index`, but the vendored paginator only has private `#index` (always `undefined → ?? 0 → chapter 0`); and a comment claimed "foliate-paginator has no setStyles" when `setStyles()` is a first-class API (`#display` wires per-doc `<style>` elements into a `WeakMap` on every section load). - **Scrolled-mode fraction is normalized over the scrollable range, not content height (2026-06-11).** Foliate's scrolled relocate fraction is `scrollTop/contentHeight`, which maxes at `1 − screen/chapterHeight` — a chapter shorter than ~20 screens could never reach the 0.95 completion threshold. `postRelocate` now reports `scrollTop/(contentHeight − screen)` (clamped; `1` for single-screen chapters), and the restore paths (`openBook`, `__foliate.goTo`) invert it via **function anchors** — foliate evaluates `anchor(doc)` after the section loads, which is the only moment `viewSize` is known. Chapter nav in scrolled mode is over-scroll: a swipe that *starts* while already pinned at the chapter boundary (same `start`/`viewSize − end` checks as foliate's own `#scrollPrev/#scrollNext`) triggers `paginator.prev()/next()`, making the advance a deliberate second gesture.

- **PDF book imports — extraction technique that actually works (2026-04-25).** The `scripts/parse-aquinas-catechetical.py` workflow is reusable for any text-based (non-scanned) Catholic-classic PDF. Three quirks worth knowing:
  - **Hybrid extraction.** `pdftotext -bbox-layout` emits XHTML where each `<block>` is a paragraph — *the* signal `pdftotext -raw` and default mode both lose. Use `-bbox-layout` for body, but `-raw` for any 2-column endnote section: `-bbox-layout` reads the columns interleaved (1, 11, 2, 13, 3, 14, …) while `-raw` linearizes them top-down per column (1, 2, 3, …, 17). Parsing endnotes off the bbox output and sorting by N is also workable but loses multi-line note continuity.
  - **Form feeds glue page numbers to text.** `pdftotext -raw` joins page-break footers to the previous token via `\x0c` — `ENDNOTES66` (note 66) and `Penance.66` style glitches. Strip `\x0c` from raw text before any line-based regex.
  - **Python 3.14 ships with a broken pyexpat on this machine** (`Symbol not found: _XML_SetAllocTrackerActivationThreshold`); ElementTree raises on import. Use `python3.13` for any XML parsing in import scripts. The shebang `#!/usr/bin/env python3.13` is the safe default for poppler-XML pipelines.
  - **PDF heading detection by ALL-CAPS works for ~80% of cases.** Inline sub-section headings buried mid-block (e.g. `"HAIL MARY"` and `"FULL OF GRACE"` quoted ALL-CAPS subheaders inside the Hail Mary chapter) survive as inline quoted phrases. They're readable but not indexed as `##`. Acceptable for v1; revisit if a future book has many of them.
  - **Scholarly editions can mix footnote conventions.** Collins's body uses `[N]` brackets; his Translator's Preface uses bare-superscript style that pdftotext renders as digits attached to a year (`year 1225.1 The` for `year 1225.[1]`). One regex (`\b\d{4}[.,]\d{1,2}\s+[A-Za-z]`) caught both instances in this PDF.

- **Explore "scroll past everything and get stuck" = RN's `automaticallyAdjustKeyboardInsets` leaking a phantom bottom inset — removed from `ScreenLayout` (2026-07-11).** TestFlight report: scroll the Explore tab to the bottom and it sails past the last row into a black void; swiping back "doesn't move" (the void is taller than a screen, so a couple of swipes still show black and it reads as frozen); re-tapping the Explore tab recovers because the native tab button issues a scroll-to-top. Root cause: `ScreenLayout`'s ScrollView set BOTH `automaticallyAdjustKeyboardInsets` and wrapped itself in react-native-keyboard-controller's `KeyboardAvoidingView` (behavior "padding") — double keyboard handling. The RN prop is buggy on the new architecture: keyboard show events (the iOS 26 search tab's integrated field is the everyday trigger — it raises the keyboard while all tab ScrollViews are mounted; rn-screens gestures can also drive it) add `contentInset.bottom`, and the matching remove is dropped for views that are off-window when the keyboard hides (facebook/react-native#47731, #41397 are this bug's family). The inset persists and can stack across keyboard sessions, so the scrollable range extends screens past the real content. Fix: keyboard avoidance is the KAV's job alone; the prop is gone from `ScreenLayout` (a comment in the file warns against re-adding it). Rule of thumb for future surfaces: never combine `automaticallyAdjustKeyboardInsets` with a KeyboardAvoidingView — pick one owner per scroll view (`PracticeEditSheet` still has the combo; it lives inside a modal that unmounts on close, which is why it hasn't bitten there).
