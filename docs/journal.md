## Architecture Decisions

- **Unified flow system replaces variants, forms, and multiple flows (2026-04-12).** The content engine accumulated four content-selection mechanisms (variants, forms, multiple flows, setKeyOverride) each built for specific practices. Decision: collapse all four into one `select` section type â€” a conditional branch that reads context (day of week, time of day, user preference) and picks an option. One flow per practice. Different meditation traditions become separate practices grouped by `alternativeTo`. Full spec: `docs/features/unified-flow-system.md`. Key insight: all four mechanisms were just conditional content selection with different indirection layers. The `variant` DB column was never actually written â€” every user was on the default.

- **DSL completeness: no practice should need custom app code (2026-04-12).** The flow JSON must be expressive enough that the app is a pure reader. The `useLiturgicalMeditation` hook (118 lines, meditacoes-ligorio only) was the clearest violation â€” calendarâ†’chapter resolution hard-coded in React. Fix: `resolve` field on `FlowDefinition` with pluggable strategies (first: `"liturgical-day"`), and dynamic `prose` sections that load book chapters by template-variable chapter IDs. The resolution algorithm stays in `@ember/liturgical` but is invoked declaratively from the flow JSON, not from a practice-specific hook.

- **Visible `select` blocks require explicit app-side rendering + override wiring (2026-04-14).** After migrating practices to unified flow, the "title-only, no body" symptom appeared only on practices whose main content lived under top-level visible `select` sections (e.g., Mass). The content engine was already emitting rendered `select` sections, but `SectionBlock` had no `select` case, so those branches rendered as `null`. Fix: add `SelectBlock` UI renderer and thread `selectOverrides` state through `PracticeFlow` so selector changes re-resolve flow content live. Silent/static patterns (e.g., Way of Light, Bible+Catechism) were unaffected.

- **Large-flow `select` switching is fast only when non-selected branches are lazy (2026-04-14).** For visible `select`, eagerly resolving sections for every option on each pass caused noticeable delay on heavy flows like Mass. The UI only needs labels for all options and sections for the active option, so the engine now resolves sections only for the selected option and emits empty section arrays for non-selected options. This keeps behavior identical while removing most of the per-toggle work.

- **Unified flow completed migration path for dynamic Liguori meditations (2026-04-14).** The old `useLiturgicalMeditation` hook was removed and replaced with declarative flow resolution: `resolveFlowAsync` executes flow-level `resolve` strategies (starting with `liturgical-day`), injects template vars, preloads dynamic prose chapter text, and then renders sections without practice-specific React logic. This made Liguori content a pure flow JSON concern and aligned slot naming previews to show the first visible select choice in Plan of Life.

- **Legacy slot flow/variant plumbing removed from runtime writes (2026-04-14).** Slot creation/update no longer writes a `variant` field, and the unused `changeSlotFlow` mutation path was removed from repositories/hooks. The database column remains for backward compatibility, but runtime behavior now treats slots as practice-scoped schedules with unified-flow selection handled by the engine.

- **Dynamic prose must support book-language fallback to avoid false-empty renders (2026-04-14).** `meditacoes-ligorio` is PT-BR-only content, so strict active-language loading can render empty prose when app/content language is EN. The fix was to make dynamic prose language behavior explicit (`langPolicy`) and preload chapter text across candidate languages (active/content/book languages), then localize from the merged payload. This keeps the flow declarative while keeping side-effect handling inside bounded source adapters.

- **Flow contract now supports explicit versioning (`flowVersion`) (2026-04-14).** The engine accepts legacy flows with no version and validates `flowVersion: "1"` for explicit-contract flows, while rejecting unsupported future versions early. This gives us room to evolve semantics safely without silent behavior drift.

- **Content loader migration phase used canonical `flow.json` preference without practice IDs (2026-04-14).** During unified-flow rollout, filesystem loading preferred top-level `flow.json` over legacy manifest declarations to avoid practice-specific branches. This transitional behavior was later tightened into strict `manifest.flow` loading once content migration was complete.

- **Legacy flow migration paths removed from filesystem loader (2026-04-14).** The temporary runtime adapters (`migrateLegacyLiturgicalMeditationFlow`, `flows[]` manifest fallback, canonical-vs-declared flow scoring) were removed. Loader contract is now strict: every practice must declare `manifest.flow`, and content was migrated accordingly.

- **Legacy Liguori sidecar flow removed (2026-04-14).** `practices/meditacoes-ligorio/flows/daily.json` was removed to avoid split truth and stale fixed-slot placeholders (`feastLabel`, `meditation-feast`). Canonical `flow.json` is the single source of truth.

- **Liguori resolver now includes monthly supplemental meditations on day 25 (2026-04-14).** `resolveLiturgicalDay` emits supplemental entries for the 25th-day monthly set, and the content-engine strategy includes those chapters in the `meditations` array. This restores expected multi-meditation days on the 25th while keeping logic in the liturgical kernel instead of app-layer hooks.

- **Day-25 rendering fix required reserve fallback for temporal holes (2026-04-14).** Some 25th dates (e.g., during Pentecost-transition gaps) had no temporal chapter in the map, so only the monthly supplemental meditation appeared. `resolveLiturgicalDay` now falls back to `reserves` when temporal lookup fails, preserving the expected â€śtemporal + monthly 25thâ€ť pairing.

- **Liguori recurrence keys now use generic `weekdaysOfMonths` + fixed-date keys (2026-04-14).** The old family keys (`first-friday/*`, `march-wednesday/*`, `25th-of-month/*`) were migrated to two generic mechanisms: fixed-date keys (`MM-DD`, e.g. `04-25`) and `weekdaysOfMonths` keys (`1st-friday-of-january`, `3rd-monday-of-february`). The resolver now computes recurrence keys directly and no longer branches on practice-specific key families.

- **Appendix map section removed in favor of `fixedDates` additive entries (2026-04-14).** Monthly 25th meditations now live in `fixedDates` directly, and the resolver merges fixed-date entries as additional content alongside temporal resolution (deduping identical chapters). The former `virtue/*` set was moved into `reserves` in declared order so it can still be used by fallback rotation without preserving practice-specific key families.

- **Resolve steps now support calendar override from JSON (`resolve[].calendar`) (2026-04-14).** The engine accepts `calendar: "ef" | "of"` on resolve steps and uses it for liturgical label generation instead of user preference when provided. Liguori flow now declares `"calendar": "ef"` in content, so EF behavior is encoded in static JSON rather than implicit runtime state.

- **54-Day Rosary Novena modeled as content-only program in a dedicated library (2026-04-13).** The implementation uses a new `novenas` library that depends on `base` for shared prayer refs, with a single practice `rosary-54-day-novena`. No app runtime changes were needed: `programDay` + `cycle` + `select` in flow JSON were enough to express petition/thanksgiving phases and the traditional Joyfulâ†’Sorrowfulâ†’Glorious 3-day cadence.

- **Novenas consolidation direction: one library for all novenas (2026-04-13).** We decided `novenas` should become the single destination for novena practices. Existing novenas in `devotions` are planned for migration first, then a broader expansion backlog (Divine Mercy, St. Joseph, St. Therese, Immaculate Conception, Christmas, and additional Marian/saint novenas) will be added in phased content-only batches.

## Infrastructure

- **Hearth replaces the broken Expo web deploy on GitHub Pages.** The original `.github/workflows/deploy.yml` built and deployed the full Expo web app to GitHub Pages but broke after the monorepo restructure (missing `baseUrl` config). Instead of fixing it, we repurposed GitHub Pages as a static content server called Hearth. It copies Bible, propers, catechism, and saints images to `_site/hearth/v1/`, converts PNGs to WebP, and generates a `manifest.json` with SHA-256 hashes. No npm install or Metro â€” runs in under a minute. The `v1/` URL prefix allows future breaking changes without breaking old app versions. Base URL: `https://ember.dpgu.me/hearth/`.

- **App fetches all large content from Hearth on demand.** Bible (DRB), Catechism (CCC), EF Mass propers, and saints images are no longer bundled in the app. They're fetched from Hearth via `apps/app/src/lib/hearth.ts` (thin `fetch()` + SQLite cache wrapper) and cached in the `cache` table. This saves ~63MB from the app bundle. Source files live in `content/` at the repo root. First launch requires network â€” accepted trade-off. The `@ember/mass-propers` package's `PropersDataSource` interface is fully async to support this.

## Content & Licensing

- **DRB is the only viable bundled English Catholic Bible.** The Douay-Rheims (1749â€“1750) is fully public domain. Every other major Catholic English translation (NABRE, RSV-CE, NJB, etc.) is under active copyright. Source: `xxruyle/Bible-DouayRheims` on GitHub, MIT license.

- **Bolls.life DRB is incomplete â€” do NOT use it.** The Bolls.life API's Douay-Rheims translation only has 66 books â€” all 7 deuterocanonical books are missing. For Catholic use, always use the bundled version from xxruyle. Non-Catholic translations (66 books) work fine but need fallback to bundled DRB for deuterocanonical content.

- **ICEL copyright blocks free OF collects and antiphons.** The Ordinary Form's Collect, Entrance/Communion Antiphons, and other variable prayers are copyrighted by the International Commission on English in the Liturgy. No free API provides them in structured format. This is a hard blocker for complete OF daily Mass propers. The EF has no such problem â€” everything is available via Missale Meum/Divinum Officium.

- **CCC usage requires Vatican attribution.** The Catechism JSON comes from scraping vatican.va. Attribution required: "Catechism of the Catholic Church, copyright Libreria Editrice Vaticana." Vatican generally permits non-commercial educational use.

- **Divinum Officium hymns use a custom bracket format.** The source files aren't JSON â€” they use `[SectionName]` brackets. Needs a one-time parse step to convert to JSON. Already done for MVP hymns.

## APIs & Integrations

- **Bolls.life API: free, no auth, generous rate limits.** Base URL: `https://bolls.life`. No documented rate limits, but don't abuse it. Responses cached indefinitely in SQLite since Bible text never changes.

- **EF propers bundled from Divinum Officium, not Missale Meum API.** Missale Meum wraps DO data but only supports English + Polish. We need Portuguese too, so we parse the raw DO files directly. The bracket-based format (`[Introitus]`, `[Oratio]`, etc.) is parsed at build time by `scripts/parse-do-propers.ts` into JSON. Portuguese coverage is partial (207 vs 480 Tempora files) â€” the app falls back to English via `localizeContent`. Cross-references (`@Tempora/...`, `@Commune/...`) point to `missa/` or `horas/` Commune directories. Prayer macros (`$Per Dominum`) expand from `Ordo/Prayers.txt`. The Commune files for Mass sections live in `horas/{lang}/Commune/` (same files serve both Office and Mass).

- **Evangelizo: OF reading text but with quirks.** `https://feed.evangelizo.org/v2/reader.php?date={YYYYMMDD}&lang={LANG}&type=all` provides full daily reading text. Limitations: response may contain HTML that needs stripping, date range limited to ~30 days from current date, no CORS headers (may need proxy for web), and does NOT provide collects/antiphons.

- **Catholic Readings API: OF calendar + references only.** GitHub Pages hosted (free, MIT). Provides Scripture references (not full text) and liturgical day metadata. The `psalm` field sometimes references non-psalm books (e.g., Jeremiah). Cross-chapter ranges use em-dash (â€”). Coverage: 2025â€“2026 data available.

- **Universalis JSONP: potentially has OF propers but unreliable.** Claims to provide full Mass propers via JSONP. Documentation is vague and incomplete. Legacy JavaScript callback pattern doesn't fit React Native well. Worth investigating further but don't count on it.

- **Bible Gateway and API.Bible are not viable.** Bible Gateway discontinued its API and prohibits scraping. API.Bible has FUMS tracking requirements implying online-only usage, unclear Catholic translation availability.

## Technical

- **ProperSlot retries share one OF query key.** Each slot on the Mass screen (Introit, Collect, Antiphon, etc.) has its own `useProperForSlot(slot, 'of')` call, but all slots read from a single `['of-propers', dateKey, language]` query. When a user taps "Try again" on any failed slot, `ofQuery.refetch()` triggers one network request and TanStack Query re-derives every slot's section from the refreshed data â€” no fan-out, no per-slot request storm. Same approach for EF but each slot has its own key (EF reads from bundled/Hearth propers, independent slots).

- **Global mutation error handler uses TanStack Query's MutationCache, not per-hook wiring.** Across 39+ mutations in 10 features, none had `onError` handlers â€” failures were silent. Rather than retrofitting each hook, `_layout.tsx` configures `QueryClient({ mutationCache: new MutationCache({ onError }) })` once. The global handler opens `ConfirmSheet` in single-action mode with `i18n.t('error.somethingWrong')` + the error message (or `error.tryAgainLater` fallback). Per-mutation `onError` still wins: the global handler guards with `if (mutation.options.onError) return`. Trade-off: every mutation error now shows a dialog unless explicitly suppressed â€” acceptable because the fallback is far better than silent failure.

- **The original `getLiturgicalSeason()` was an antiphon scheduler, not a season calculator.** Its boundaries mapped to the Marian antiphon switching dates (Adventâ†’Feb 1, Feb 2â†’Holy Wednesday, etc.), which don't correspond to actual liturgical seasons. The antiphon schedule is a separate traditional system with its own date ranges â€” decoupled from seasons in the rewrite.

- **OF and EF seasons share Easter and Advent but diverge everywhere else.** Key differences: EF has Epiphanytide (Jan 14 to Septuagesima) and Septuagesimatide (3 pre-Lent Sundays using violet), neither of which exist in OF. EF has "Time after Pentecost" where OF has "Ordinary Time." Christmas ends at Baptism of the Lord (OF, ~Jan 11) vs Octave of Epiphany (EF, Jan 13). A single `LiturgicalSeason` union with 8 values covers both forms cleanly â€” each form uses a subset.

- **Year-boundary edge case in season calculation.** Dec 25 is after the start of Advent, so a naive "if date >= Advent start, return advent" incorrectly classifies Christmas Day. The fix: check Christmas (Dec 25+) before checking Advent. Similarly, Jan 1 is still Christmastide in both forms.

- **Divine Office migrated to practice content format.** The custom engine (`divine-office/engine.ts`) and `PrayerFlow` component were replaced by JSON flow files + the shared content engine. Three key infrastructure changes accompanied this: (1) `practice_completions` event log replaces `practice_logs` â€” supports multiple completions per day and per-hour detail tracking; (2) `reading_tracks` replaces `reading_progress` â€” named cursors with `id` PK instead of `type` PK, enabling future track sharing between practices; (3) reading tracks are auto-advanced on practice completion whenever `tracks` are defined in the manifest. The `daily_office` and `practice_logs` tables still exist but are deprecated.

- **Unified hours/forms/flow into a single `flows` model.** Three previously separate manifest fields (`flow`, `hours`, `forms`) were replaced by a single `flows: FlowEntry[]` array. Each flow is a named, schedulable prayer sequence. Scheduling (tier, day-of-week) moved from the `Variant` type into `defaults.slots[]` in the manifest â€” each `SlotDefault` maps a `flowId` to a schedule. The `Variant` type was simplified: `selector`, `schedule`, and `setNames` fields were removed; the engine now receives the active flow ID as `setKeyOverride` in `FlowContext` and picks `variant.data[flowId]` directly (falls back to the first key). This made the Rosary manifest explicit: 4 flows (joyful/sorrowful/glorious/luminous) with day-of-week slots, instead of an opaque `selector: "day-of-week"` schedule inside the variant. Simple practices get a single flow with `id: "default"`.

- **Tamagui sub-theme nesting for liturgical seasons.** Wrapping the app in `<Theme name={season}>` inside `<TamaguiProvider defaultTheme={resolvedTheme}>` causes Tamagui to resolve `light` + `advent` to the registered `light_advent` theme automatically. This means any component reading `$accent` gets the seasonal color with zero changes to those components. Important: `useThemeName()` now returns the sub-theme name (e.g., `'dark_lent'`), not just `'dark'` â€” any code checking theme name equality must use `.startsWith('dark')` instead of `=== 'dark'`.

- **Backward-compatible wrappers for reading tracks.** `getReadingProgressByType('ot')` maps to `getReadingTrack('default-ot')` so existing consumers (settings screens, reading progress hooks) work without changes. The `ReadingTrack` type is a superset of `ReadingProgress` (adds `id` and `label`).

- **Rubrics restyled from section headers to Missal-style italic.** The `RubricLabel` component was originally uppercase, letter-spaced Cinzel text â€” functioning as a section header. Real Missal rubrics are italic, sentence case, in red (hence "rubric" from Latin *rubrica*). The component was restyled to italic body font with accent color. All 40+ flows that used rubrics as section labels ("Opening Verse", "Hymn", "Psalmody") were migrated to `subheading` type instead. Mass flows received comprehensive rubrics at every major gesture (genuflections, bows, signs of the cross, hand positions) following the 1962 Missale Romanum (EF) and GIRM (OF).

- **Mass view modes use grouped flows, not engine filtering.** We considered two approaches for Propers/Readings views: (1) a single flow with engine-level section filtering by `viewMode` in `FlowContext`, or (2) separate flow JSON files with a `group` field on `FlowEntry` linking sub-flows to their parent. We chose (2) because it's zero engine complexity â€” just JSON content + a `group` field. The `ViewModeSelector` component renders a segmented control inside the pray screen; switching modes swaps the loaded flow via `useState` without navigation. The `FlowButtons` component on the detail screen filters out grouped flows so only primary flows appear as entry points.

- **OF propers: Liturgia DiĂˇria API gives complete PT-BR propers, Evangelizo gives EN readings only.** The `liturgia.up.railway.app/v2/` API (from `Dancrf/liturgia-diaria` on GitHub) provides all OF Mass propers in Portuguese â€” collects, prayers, antiphons, and readings with full text. Found via `servusdei2018/awesome-catholic`. For English, only readings are available (Evangelizo) â€” ICEL copyright blocks English collects/antiphons. Universalis was investigated thoroughly and confirmed to only serve readings despite vague docs. See `docs/content/of-propers-research.md` for full research.

- **Evangelizo has no psalm response/refrain.** The API returns only the psalm verses â€” the responsorial refrain (e.g., "Keep me, O God, for in you I take refuge") is not included. The Liturgia DiĂˇria API provides this for PT-BR via the `refrao` field, but no structured English source has been found. This is a known English data gap.

- **Evangelizo response format is HTML, not JSON.** The `feed.evangelizo.org` API returns HTML with `<br />` tags, `&quot;` entities, and `<font>` tags for citations. Sections are separated by triple `<br />` tags. Needs HTML stripping and section detection based on content (Psalms â†’ psalm slot, Gospel â†’ gospel slot, etc.).

- **Proper form must come from flow JSON, not user preferences.** A user may prefer the EF liturgical calendar but still be viewing the OF Mass practice. The `form` field on `proper` sections in flow JSON determines which data source is used. This is passed through the content engine to the `ProperSlot` component.

- **EF Mass propers: Tempora/Sancti precedence must use the liturgical calendar.** A naive merge (Sancti overlays Tempora) is wrong â€” on Holy Thursday 2026, St. Francis of Paola (Sancti `04-02`, III class) was overriding the Triduum epistle with an empty `Lectio`. The fix: use `DayCalendar.principal` from `buildYearCalendar()` to determine which source provides the Mass. If the principal's `category` is temporal (`solemnity_temporal`, `feast_of_the_lord`, `liturgical_season`) â†’ use Tempora propers. If it's a saint category â†’ use Sancti propers. Never merge both. Fallback without a calendar: Tempora-only (safe default â€” missing a saint is less wrong than overriding the Triduum). Future: commemorated saints should contribute only their Collect (Oratio) to the Mass of the day.

- **Friday abstinence rules differ significantly by jurisdiction.** Universal canon law (Canon 1251) requires meat abstinence on ALL Fridays year-round. However, the USCCB (1966 Pastoral Statement) replaced obligatory Friday abstinence outside Lent with a general penance obligation (any form). The CNBB (Brazil) kept year-round Friday abstinence but allows substitution with charity or other food â€” subtly different from the US approach. EF (1962 rubrics) maintains complete abstinence on all Fridays plus vigil days. Modeled as `AbstinenceLevel: 'full' | 'partial' | 'penance-required' | 'none'` in `packages/liturgical/src/obligations.ts`.

- **Books are raw HTML/markdown chapters inside `.pray`, not EPUB files.** The original plan was to embed `.epub` files in `.pray` archives and use epub.js or Readium for rendering. This was simplified: book chapters are raw `.html` or `.md` files bundled directly in the `.pray` zip under `books/{bookId}/{lang}/`. The app reads them from disk via `expo-file-system` and renders in a WebView with CSS column pagination. Markdown files (`.md`) are converted at runtime using `marked` + `marked-footnote` (~45KB). This eliminated the pandoc build dependency, JSZip from the reader path, and the double-unzip problem (.pray ZIP containing .epub ZIP). Content structure: `content/libraries/{libraryId}/books/{bookId}/{lang}/`.

- **Ember days in calendar-data.ts are placeholders, not properly resolved.** The `ember-days` entry has a dummy fixed date (Jan 1) and `ember-days-advent` only resolves to the Advent Ember Wednesday. The obligations module computes all 12 ember days (4 seasons Ă— Wed/Fri/Sat) from scratch using anchor dates. Future: consider moving ember day resolution into the calendar builder.

- **Practice-flow readings and psalmody surface inline retry on fetch failure (2026-04-18).** `BibleReadingBlock` / `CccReadingBlock` return `undefined` when data is missing, and `usePsalmsForHour` previously filtered to successful results, so an errored Bible / CCC / psalm fetch disappeared silently from the composed practice flow (no empty, no error â€” just a gap). Fixes: `PracticeFlow` tracks `bibleErrors` / `cccErrors` maps of `queryKey â†’ refetch()` alongside the data maps and renders a small inline retry card at the reading's position; `usePsalmsForHour` now returns `slots: PsalmSlot[]` (per-ref `{verses?, retry?}`) and `PsalmodyBlock` renders the same retry at each failed psalm's position. `isDynamicLoading` already falls through on errors (it only checks `isLoading`), so the inline state is reached without extra gating. Reused existing `common.couldntLoad` / `common.retry` strings (same keys used by `ReaderErrorState`) via a shared `InlineRetry` component â€” the full-screen `ReaderErrorState` uses `flex={1}` and wouldn't compose inside a section list, so `InlineRetry` is a separate, slim variant.

- **Base library expanded with catechetical chapters â€” creeds, commandments, sacraments, virtues (2026-04-19).** The `base` library (previously named `ember-default`) was broadened from "prayers + practices" to "the essentials of the Christian life." Added 8 reference chapters: Ten Commandments, Beatitudes, Precepts of the Church, Seven Sacraments, Corporal/Spiritual Works of Mercy, Virtues (theological + cardinal), Capital Sins and Opposing Virtues, Gifts and Fruits of the Holy Spirit. Also added the Nicene-Constantinopolitan Creed (EN/PT-BR/Latin) alongside the Apostles' Creed. Design choices: (1) each chapter is a short prose reference â€” not a practice â€” following the `sacred-heart/chapters/twelve-promises` pattern (3â€“4 sections: intro â†’ body â†’ optional second body, divided by simple line); (2) estimated minutes kept to 7â€“8, aimed at a single reading; (3) tone is catechetical without being encyclopedic â€” formative, quotes CCC and Scripture only when it adds weight; (4) no flow DSL required (chapters are static prose). Concurrently renamed all `ember-*` libraries to unbranded names (`ember-default`â†’`base`, `ember-extra`â†’`devotions`, `ember-novenas`â†’`novenas`) â€” branding is owned by the app, not by content.
