# Internationalization (i18n)

## Overview

Ember uses [react-i18next](https://react.i18next.com/) with [expo-localization](https://docs.expo.dev/versions/latest/sdk/localization/) for multi-language support. Currently supported: English (`en`) and Brazilian Portuguese (`pt-BR`).

## Architecture

- **Bundled translations**: All strings are compiled into the app — no network requests. Works fully offline.
- **Single namespace**: One `translation` namespace with ~150 keys, organized by screen/feature.
- **Synchronous init**: `i18n.init()` runs at import time (`src/lib/i18n/index.ts`), before the app renders.
- **Language persistence**: Stored in AsyncStorage via `preferencesStore.language`. Auto-detects device locale on first launch.

## Key Files

```
src/lib/i18n/
  index.ts            — i18next config, supportedLanguages export
  dateLocale.ts       — formatLocalized() for locale-aware dates
  locales/
    en.ts             — English translations
    pt-BR.ts          — Brazilian Portuguese translations
```

## How Translations Work

### In React components

```typescript
import { useTranslation } from 'react-i18next'

function MyComponent() {
  const { t } = useTranslation()
  return <Text>{t('home.greetingMorning')}</Text>
}
```

### In pure functions (non-components)

```typescript
import i18n from '@/lib/i18n'

function getLabel(): string {
  return i18n.t('office.morningPrayer')
}
```

### Interpolation

```typescript
t('settings.booksOf', { completed: 5, total: 73 })
// → "5 of 73 books" or "5 de 73 livros"
```

### Date formatting

Use `formatLocalized` instead of `format` from date-fns for user-visible dates:

```typescript
import { formatLocalized } from '@/lib/i18n/dateLocale'

formatLocalized(new Date(), 'EEEE, MMMM d')
// → "Thursday, March 27" or "quinta-feira, março 27"
```

Keep `format` for machine-readable dates (`'yyyy-MM-dd'`).

## Translation Key Structure

Keys are organized by screen/feature:

| Prefix | Area |
|--------|------|
| `home.*` | Home screen — greetings, nav labels |
| `office.*` | Divine Office — hour labels, action buttons |
| `rubric.*` | Prayer rubric labels (Opening Verse, Hymn, etc.) |
| `plan.*` | Plan of Life — title, stats, empty states |
| `editor.*` | Practice edit form labels |
| `frequency.*` | Frequency picker labels |
| `timeBlock.*` | Time block labels (Morning, Daytime, etc.) |
| `tier.*` | Practice tier labels (Essential, Ideal, Extra) |
| `day.*` | Day abbreviations (Sun, Mon, etc.) |
| `settings.*` | Settings screen labels |
| `readingLabel.*` | Reading type labels (OT, NT, Catechism) |
| `books.*` | Mark Books as Read screen |
| `position.*` | Reading position picker |
| `bible.*` | Bible reader labels |
| `catechism.*` | Catechism reader labels |
| `translations.*` | Translation picker labels |
| `readingConfig.*` | Reading config modal labels |
| `common.*` | Shared labels (Loading, Back) |
| `practice.*` | Builtin practice names (16 total) |

## Practice Names

The 16 builtin practice names are translated via i18n keys (`practice.morning-offering`, etc.), not stored translated in the database. Custom user-created practices use their stored name directly.

## Prayer & Hymn Content

Prayer and hymn JSON files use a multi-key structure:

```json
{
  "title": "Our Father",
  "latin": "Pater noster...",
  "english": "Our Father, who art in heaven...",
  "portuguese": "Pai nosso, que estais nos céus..."
}
```

The Divine Office engine selects the appropriate key based on the current language.

## Adding a New Language

1. Create `src/lib/i18n/locales/{code}.ts` with translations for all keys
2. Register it in `src/lib/i18n/index.ts` resources
3. Add to `supportedLanguages` array
4. Add date-fns locale mapping in `src/lib/i18n/dateLocale.ts`
5. Add `{code}` translations to prayer JSON files (`src/assets/prayers/*.json`)
6. Add `{code}` translations to hymn JSON files (`src/assets/hymns/*.json`)
